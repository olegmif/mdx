# M4 — поиск по тегам

## Цель

После M4 в markdown-буфере нажатие `<leader>mt` (или выполнение `:MdxTagSearch`) открывает Telescope-picker со списком всех заметок и окном превью справа. По мере того, как пользователь печатает в prompt'е токены вида `tag1 tag2* -tag3 -*tmp* t?g`, picker сужает список заметок: учитываются только те, у которых есть **все** теги, удовлетворяющие включениям, и **нет ни одного**, удовлетворяющего исключениям. Каждый токен — либо точное имя тега, либо паттерн с wildcard'ами (`*` — любая последовательность символов; `?` — ровно один символ). Выбор заметки открывает её в текущем буфере (как `:MdxFollow`).

После M4 типовой цикл «увидеть заметки по тегу → уточнить набор тегов → открыть нужную» закрыт целиком в neovim, поверх **всей файловой системы**, на индексе тегов, который наполняет M0 (`mdx scan`) и поддерживает M1 (LSP upsert на open/save). Транспортный слой `mdx/*` расширяется одним кастомным методом по той же схеме, что введена в M3.

Discoverability тегов решается через окно превью: пользователь видит начало каждой заметки, включая `tags:` во frontmatter, и из этого соображает, какими токенами сузить или расширить выборку. Отдельный picker / справочник тегов в M4 не вводится.

## Что входит и что не входит

### Входит

- Кастомный LSP-метод `mdx/searchByTags` на стороне Go-сервера: принимает параметр `{ include: []string, exclude: []string }`, возвращает массив записей `{ path, title }` для заметок, у которых каждый паттерн из `include` находит хотя бы один тег и ни один паттерн из `exclude` не находит ни одного. Паттерн без GLOB-метасимволов — точное имя тега. При обоих пустых массивах — все заметки.
- Функция `db.SearchByTags(db, include, exclude)` в Go-слое БД: динамически собранный SQL-запрос по таблицам `notes` и `tags` через per-token `EXISTS`/`NOT EXISTS`. Выбор условия per-token — `tag = ?` для точного имени, `tag GLOB ?` для паттерна с любым из GLOB-метасимволов (`*`, `?`, `[`).
- Lua-функция `tags.parse_query(prompt) -> include, exclude` в `lua/mdx/tags.lua`: чистая функция-парсер prompt'а в два массива токенов (точные имена и `*`-паттерны различает Go-сторона, Lua только нарезает по пробелам и снимает `+`/`-`). Незавершённый последний токен (без trailing space) игнорируется.
- Lua-функция `tags.search(include, exclude)` в том же файле: синхронный JSON-RPC-вызов к подключённому `mdx`-LSP-клиенту, возвращает массив записей или `nil`.
- Picker `picker.tag_search(opts, on_select)` в `lua/mdx/picker.lua`: Telescope с `finders.new_dynamic`, источник пересоздаётся на каждое изменение prompt'а через `tags.parse_query` + `tags.search`. Превью — стандартный `file_previewer`.
- Оркестратор `M.tag_search()` в `lua/mdx/init.lua`: открывает picker, по выбору открывает заметку через `vim.cmd.edit` (с экранированием пути через `vim.fn.fnameescape`).
- Команда `:MdxTagSearch`, keymap `<leader>mt` (default), регистрация в `setup()` и `ftplugin/markdown.lua`.
- Расширение `defaults.keymaps` полем `tag_search = "<leader>mt"`.
- Go-тесты на `db.SearchByTags` и LSP-метод `mdx/searchByTags`.
- Lua-тесты: табличные на `tags.parse_query`, smoke на `tag_search` через подмену `picker.tag_search`.

### Не входит (отложено)

- Открытие выбранной заметки в split — кандидат на M4.5 (по аналогии с парой `MdxFollow`/`MdxFollowSplit`).
- OR между тегами (логическое ИЛИ). Грамматика prompt'а — только AND-includes и NOT-excludes; для `(a OR b)` потребовался бы синтаксис со скобками или альтернативным разделителем, что усложняет парсер ради редкого паттерна.
- Fuzzy-фильтр по заголовку внутри того же picker'а. Все токены prompt'а интерпретируются как имена тегов; для поиска по заголовку у пользователя есть `:MdxInsertLink` из M3 (как браузер заметок) и в будущем — отдельная команда поиска.
- Автоматическое разворачивание иерархии тегов: тег `a/b/c` хранится и сравнивается как одна строка, не разворачивается в `a`, `a/b`, `a/b/c`. (Псевдо-иерархия достижима пользовательским паттерном — `a/*` подцепит все «вложенные».)
- Отдельный справочник тегов / счётчиков (`mdx/listTags`). Discoverability — через превью.
- Tag completion в prompt'е picker'а (cmp-source / floating popup) — самостоятельный сюжет.
- LSP completion для тегов в самом markdown-буфере (при наборе `#` или внутри frontmatter `tags:`).
- Conceal или подсветка тегов в буфере.
- Создание новой заметки с предзаданным тегом из picker'а.
- Альтернативные picker-движки (`fzf-lua`, `snacks.picker`, `vim.ui.select`) — поддерживается только Telescope.

## Технические решения

| Вопрос | Решение |
|---|---|
| Сценарий выбора | Один picker, одна команда. Prompt — DSL над тегами. Двух-фазный «picker тегов → picker заметок» отвергнут как избыточный (две команды, два окна, лишний round trip ради того, что и так видно в превью первого picker'а). |
| Источник заметок | Кастомный LSP-метод `mdx/searchByTags`. Параметр — `{ include: []string, exclude: []string }`. Возвращает `[]NoteEntry` (тот же тип, что и `mdx/listNotes`). Сортировка `title COLLATE NOCASE, path` — на стороне Go. |
| Имя метода | `mdx/searchByTags`. Префикс `mdx/`, camelCase — соответствует конвенции `mdx/*`, заданной в M3. |
| Чем `mdx/searchByTags` лучше пары `mdx/listTags` + `mdx/notesByTag` | UX выбран как «сразу заметки + уточнение через prompt», поэтому отдельный picker тегов не нужен; справочник `mdx/listTags` без UI-подложки тоже не нужен. Один метод покрывает все состояния picker'а (пустой prompt → все заметки; добавился token → сужение). |
| Чем `mdx/searchByTags` лучше `workspace/symbol` | `workspace/symbol` рассчитан на символы кода с обязательным `Location`. У нас выборка по тегам — это `JOIN` двух таблиц с группировкой; натянуть это на семантику символов нельзя без потери поля и без фиктивных Location'ов. |
| Грамматика prompt'а | Токены, разделённые пробелами. Префикс `+` — необязателен (по умолчанию include). Префикс `-` — exclude. Любой из символов `*`, `?`, `[` внутри токена — wildcard (см. ниже). Незавершённый последний токен (нет trailing space) игнорируется парсером. Пустой prompt — пустые `include` и `exclude`, сервер возвращает все заметки. |
| Wildcards и точное совпадение | Токен **без** GLOB-метасимволов сравнивается через `tag = ?` (точное совпадение, использует индекс `idx_tags_tag`). Токен **с** хотя бы одним из `*`, `?`, `[` сравнивается через `tag GLOB ?`: `*` — любая последовательность символов; `?` — ровно один символ; `[abc]` / `[a-z]` — character class. Префиксный `tag*` использует индекс; всё остальное (`*tag`, `*tag*`, `t?g`, `tag[12]`) идёт полным сканом таблицы `tags` — на реалистичных размерах (десятки–сотни тысяч строк) это микросекунды-миллисекунды, в практическом фидбеке picker'а незаметно. |
| GLOB вместо LIKE | LIKE в SQLite по умолчанию case-insensitive и не использует индекс для префиксных паттернов без `PRAGMA case_sensitive_like = ON`. GLOB — case-sensitive, использует индекс для префиксных паттернов, и его метасимволы (`*`, `?`, `[...]`) — те самые, которых интуитивно ожидает пользователь от wildcard-синтаксиса. Парсер тегов M0 разрешает только `[\w][\w/-]*`, поэтому ни `*`, ни `?`, ни `[`, ни `]` никогда не встречаются в реальных именах тегов: их появление в prompt'е однозначно означает намерение использовать wildcard, и escape-логика не нужна. |
| Форма SQL | Per-token `EXISTS`/`NOT EXISTS` через AND, не `IN(...) GROUP BY HAVING COUNT(...)`. Такая форма естественно расширяется на смесь точных и wildcard-паттернов (counting distinct tag values для wildcard-паттернов даёт неверную семантику — заметка с тремя `mdx-*`-тегами «совпадает с паттерном `mdx*` один раз», а не три). |
| Почему игнорируем незавершённый токен | На каждый keystroke picker дёргает finder; если применять «полу-набранный» токен `m`, картинка шарахается между «нет совпадений (тега `m` нет)» и «есть совпадения (`mdx`)». Применение по trailing space — обычная shell-семантика и совпадает с моментом, когда пользователь сам считает токен законченным. |
| Дискаверабельность тегов | Через окно превью: видно начало каждой заметки, включая `tags:` во frontmatter. Отдельный список тегов или их completion в M4 не делается. Если на практике окажется недостаточно — добавится отдельной командой (`:MdxTagList`) в M4.5. |
| Источник истины для тегов | Таблица `tags(path, tag)`, наполняется на M0/M1. M4 не пересчитывает теги. |
| Чувствительность к регистру тега | И `tag = ?`, и `tag GLOB ?` — case-sensitive. Парсер тегов M0 не нормализует регистр — теги хранятся как написаны, поэтому case-sensitive поведение совпадает с ожиданием. |
| Иерархия | Автоматически не разворачивается. Тег `proj/mdx` — обычная строка; искомые «дочерние» теги достижимы паттерном `proj/*`. |
| Открытие заметки | `vim.cmd.edit` (как `M.follow`), не split. Для split — отдельная команда (отложено). |
| Sync vs async LSP-запрос | Sync с `timeout_ms = 2000` на каждый вызов `tags.search`. Дёргается на каждое изменение prompt'а; БД маленькая, запрос мгновенный. Если на практике появятся видимые лаги — добавим debounce в Lua перед `request_sync` (отложено до сигнала). |
| Default keymap | `<leader>mt` в markdown-буферах. Регистрируется в `ftplugin/markdown.lua` на основе `M.config.keymaps.tag_search`. |
| Скоуп активации команды | `:MdxTagSearch` доступна везде после `setup()`, но требует подключённого `mdx`-LSP-клиента. Если клиент не найден — `vim.notify("mdx: LSP client not attached", ERROR)`, picker не открывается (та же модель, что у `:MdxInsertLink`). |
| Поведение при пустой выборке | Picker остаётся открытым с пустым списком — пользователь правит prompt и состояние пересчитывается. Никаких уведомлений не выводим: ноль результатов в picker'е — это и есть фидбек. |
| Picker engine | Telescope. Зависимость уже объявлена в lazy-spec из M3. |
| Динамический title с числом результатов | Вне обязательного скоупа. Если Telescope-API позволяет обновлять `results_title` без рекреации picker'а — реализуем как часть Шага 5; иначе оставляем статический title `mdx: tag search`. |
| Тестовая стратегия Go | `db.SearchByTags` тестируется на in-memory SQLite со предзаполненными `notes`/`tags`: пустые фильтры → все заметки; только `include` → AND-семантика; только `exclude` → исключение; смешанный случай; теги, отсутствующие в БД, — пустая выборка. LSP-метод тестируется через расширение `lsp/internal/lsp/e2e_test.go` (или `handlers_test.go`). |
| Тестовая стратегия Lua | `tags.parse_query` — табличные кейсы. `tag_search` — smoke-тест через подмену `require("mdx.picker").tag_search`: подмена немедленно вызывает `on_select({ path, title })`, проверяется `vim.api.nvim_buf_get_name(0)`. `tags.search` сама не тестируется (мокировать LSP-клиент в headless-плёнэри неоправданно сложно — та же позиция, что в M3 для `notes.lua`). Telescope-UI не тестируется. |

case "mdx/searchByTags":

        Include []string `json:"include"`
        Exclude []string `json:"exclude"`

    if err := json.Unmarshal(ctx.Params, &p); err != nil {

### Шаг 2. LSP-метод `mdx/searchByTags`

Реализовать выборку из БД и серверный handler.

#### Состав `lsp/internal/db/queries.go`

`SearchByTags(db, include, exclude)` динамически собирает SQL по схеме per-token EXISTS/NOT EXISTS:

```sql
SELECT path, COALESCE(title, '') AS title
FROM notes
WHERE
  -- по одному EXISTS на каждый include-токен:
  EXISTS (SELECT 1 FROM tags WHERE path = notes.path AND tag = ?)        -- exact
  AND EXISTS (SELECT 1 FROM tags WHERE path = notes.path AND tag GLOB ?) -- wildcard
  -- по одному NOT EXISTS на каждый exclude-токен:
  AND NOT EXISTS (SELECT 1 FROM tags WHERE path = notes.path AND tag = ?)
ORDER BY (COALESCE(title, '') = '') ASC,
         title COLLATE NOCASE,
         path
```

Реализация через `strings.Builder`. Для каждого include/exclude-токена:

1. Если в токене есть хотя бы один из `*`, `?`, `[` (`strings.ContainsAny(token, "*?[")`) — условие `tag GLOB ?`, аргумент — токен как есть.
2. Иначе — условие `tag = ?`, аргумент — токен как есть.

Готовая фраза заворачивается в `EXISTS (SELECT 1 FROM tags WHERE path = notes.path AND <cond>)` (для include) или `NOT EXISTS (...)` (для exclude). Все фразы соединяются через ` AND ` и подставляются после `WHERE`.

Уместно вынести выбор условия в маленькую `func tagCondition(token string) (sqlFragment, arg string)` — две ветки `=` / `GLOB`. Это упрощает Go-тесты на саму dispatch-логику.

При обоих пустых slice'ах — `WHERE` опускается, возвращаются все заметки. При пустой таблице `notes` — пустой массив, не `nil`.

Эскейп GLOB-метасимволов в литеральной части токена не делается: парсер тегов M0 (`[\w][\w/-]*`) исключает `*`, `?`, `[`, `]` из реальных имён тегов, поэтому появление этих символов в prompt'е однозначно означает wildcard. Зависимость от инварианта парсера явно зафиксирована в комментарии у `tagCondition`.

#### Состав `lsp/internal/lsp/handlers.go`

`onSearchByTags(ctx, include, exclude)`:

1. Берёт `s.conn`.
2. Вызывает `db.SearchByTags(s.conn, include, exclude)`.
3. Подставляет basename без `.md` в `entries[i].Title`, если он пустой (точно так же, как в `onListNotes` из M3 step 7).
4. Пересортировывает результат после fallback'а: case-insensitive по `Title`, потом по `Path`.
5. Возвращает результат.

Регистрация метода в `mdxHandler.Handle` уже выполнена в Шаге 1.

#### Тесты

`lsp/internal/db/queries_test.go` — добавить кейсы для `SearchByTags`:

Точные совпадения:

- Пустая БД → пустой массив.
- Пустые `include` и `exclude` → все заметки в правильном порядке.
- Один тег в `include` → заметки только с этим тегом; заметка с близким именем (`mdx-design` при поиске `mdx`) не проходит.
- Два тега в `include` → только заметки, у которых есть оба (заметка с одним из тегов не проходит).
- Только `exclude` → все заметки кроме помеченных.
- Смешанный случай: `include = ["mdx"]`, `exclude = ["draft"]`.
- Тег, отсутствующий в БД, в `include` → пустой массив.
- Тег, отсутствующий в БД, в `exclude` → не сужает выборку.

Wildcards (`*`):

- Префикс `mdx*` в `include` → заметки с тегами `mdx`, `mdx-design`, `mdx/lsp` и т.п.; заметка только с `vim` не проходит.
- Суффикс `*tmp` в `include` → заметки с тегами, заканчивающимися на `tmp`.
- Обоюдный `*draft*` в `exclude` → исключаются заметки с любым тегом, содержащим `draft`.
- Смешанный wildcard + exact: `include = ["go", "mdx*"]` → заметки, у которых есть тег `go` И хотя бы один тег по паттерну `mdx*`. Заметка с двумя `mdx-*`-тегами засчитывается так же, как заметка с одним.
- Одиночный `*` в `include` → все заметки, у которых есть хотя бы один тег.
- Одиночный `*` в `exclude` → заметки без тегов вообще.
- Конфликтующая пара: `include = ["mdx*"]`, `exclude = ["mdx-draft"]` → заметки с любым `mdx*`-тегом, кроме точно `mdx-draft`; заметка с тегами `mdx-foo` и `mdx-draft` исключается, заметка только с `mdx-foo` проходит.

Wildcards (`?`, `[...]`):

- Single-char `t?g` в `include` → заметки с тегом длины 3, начинающимся на `t` и заканчивающимся на `g` (`tag`, `tog`, `tug`).
- Комбинация `t?g*` → заметки с тегами, начинающимися на любую из `tag`/`tog`/`tug` и продолжающимися произвольно.
- Character class `tag[12]` в `include` → заметки с тегами `tag1` или `tag2`, но не `tag3`.

`lsp/internal/lsp/handlers_test.go` или `e2e_test.go` — JSON-RPC-вызовы `mdx/searchByTags` с разными параметрами; проверка структуры ответа и fallback'а пустого `title` на basename.

**Готовность:** все Go-тесты зелёные. Ручная проверка: ручной JSON-RPC-запрос к `mdx-lsp` возвращает корректный JSON.

### Шаг 3. Парсер `tags.parse_query`

Реализовать чистую функцию-парсер до подключения LSP — её удобно покрыть табличными тестами заранее.

#### Состав `lua/mdx/tags.lua`

```lua
function M.parse_query(prompt)
```

Поведение:

1. Если `prompt` — `nil`, пустая строка или только whitespace, возвращает `{}, {}`.
2. Запоминает, заканчивается ли `prompt` пробелом (`trailing_space`).
3. Триммит, бьёт по `%s+` через `vim.split` → `parts`.
4. Если `not trailing_space` — отбрасывает последний элемент `parts` (незавершённый токен).
5. Для каждого токена:
   - Первый символ `-` → остаток (после первого символа) добавляется в `exclude` (если не пустой).
   - Первый символ `+` → остаток в `include` (если не пустой).
   - Иначе → весь токен в `include` (если не пустой).
6. Возвращает `include, exclude`.

#### Тесты

`tests/tag_search_spec.lua` — табличные кейсы на `parse_query`:

|---|---|---|
| `""` | `{}` | `{}` |

| `"mdx"` | `{}` | `{}` (последний токен не завершён) |
| `"mdx "` | `{"mdx"}` | `{}` |
| `"mdx go "` | `{"mdx", "go"}` | `{}` |
| `"mdx +go "` | `{"mdx", "go"}` | `{}` (`+` синонимичен голому токену) |
| `"mdx -draft "` | `{"mdx"}` | `{"draft"}` |
| `"-draft "` | `{}` | `{"draft"}` (только exclude — допустимо) |

| `"mdx + "` | `{"mdx"}` | `{}` (одиночный `+` после расщепления — пустой остаток, отбрасывается) |
| `"mdx - "` | `{"mdx"}` | `{}` (то же для одиночного `-`) |
| `"mdx* "` | `{"mdx*"}` | `{}` (wildcard сохраняется в токене) |

### Шаг 4. Lua-обёртка `tags.search`

Связать Lua с серверным методом.

#### Состав `lua/mdx/tags.lua`

```lua
function M.search(include, exclude)
```

Поведение (структурно повторяет `notes.list()` из M3):

1. Получает первый клиент `mdx`-LSP через `vim.lsp.get_clients({ name = "mdx" })`. Если клиента нет — `vim.notify("mdx: LSP client not attached", ERROR)` и `nil`.
2. `client:request_sync("mdx/searchByTags", { include = include, exclude = exclude }, 2000, 0)`.
3. На отсутствие ответа (timeout) — `vim.notify("mdx: searchByTags timed out", ERROR)` и `nil`.
4. На `response.err ~= nil` — `vim.notify("mdx: searchByTags error: " .. msg, ERROR)` и `nil`.
5. Возвращает `response.result` (массив `{ path, title }`).

`include`/`exclude` — обычные Lua-таблицы; пустые `{}` сериализуются `vim.json.encode` как `{}` (объект), сервер на стороне Go трактует это как nil-slice через `json.Unmarshal` — поведение совпадает с пустым массивом.

**Готовность:** `:lua print(vim.inspect(require('mdx.tags').search({"mdx"}, {})))` в открытом markdown-буфере выводит массив записей. На отсутствие LSP-клиента — уведомление-ошибка и `nil`.

### Шаг 5. Picker `tag_search` и оркестрация `M.tag_search`

```lua
function M.tag_search(opts, on_select)
```

Поведение:

1. `pcall(require, "telescope.pickers")` — если `false`, `vim.notify("mdx: telescope.nvim is required", ERROR)` и выход.

       end,
       entry_maker = function(entry)
           local display_path = insert.to_display_path(entry.path)

### Шаг 7. Документация

Дописать `nvim/README.md`:

- Команда `:MdxTagSearch`, поведение, требование подключённого `mdx`-LSP-клиента.
- Keymap `<leader>mt` по умолчанию.
- Грамматика prompt'а: `tag` или `+tag` — include, `-tag` — exclude, незавершённый последний токен игнорируется до пробела. Все включения соединяются AND, все исключения — NOT. OR не поддерживается.
- Wildcards: `*` — любая последовательность символов; `?` — ровно один символ. `mdx*` — все теги, начинающиеся на `mdx`; `*tmp` — заканчивающиеся на `tmp`; `*draft*` — содержащие `draft`; `t?g` — теги длины 3 вида `t?g` (полезно, если не помнишь, через какую гласную записал тег). Точное совпадение — токен без wildcard'ов. Регистр учитывается. Псевдо-иерархия: `proj/*` подцепит все «вложенные» теги.
- Указание, что источник — БД, наполняемая `mdx scan` и LSP'ом на open/save. Теги извлекаются из frontmatter (`tags:` строкой через запятую или списком) и из `#tag`-вхождений в теле; иерархия не разворачивается (`a/b` — один тег).
- Сценарий «открыть picker → ввести тег → посмотреть в превью frontmatter других заметок → уточнить токенами → открыть нужную» как пример замкнутого цикла.
- Пометка: для просмотра выбранной заметки в split — отложено (см. M4.5).

**Готовность:** README покрывает M4-функциональность так же, как покрывает M2/M3; «как искать заметки по тегу» — один экран.

## Критерии готовности M4

1. `go test ./...` в `lsp/` зелёный, включая новые тесты на `db.SearchByTags` и `mdx/searchByTags`-handler.
2. `nvim --headless` с `PlenaryBustedDirectory nvim/tests/` зелёный, включая новый `tag_search_spec.lua` (парсер + smoke).
3. После `mdx scan ~/projects/mdx/` БД содержит теги из заметок проекта; в открытом `.md`-файле `<leader>mt` открывает picker с полным списком заметок и непустым превью.
4. Ввод `<имя_существующего_тега><пробел>` сужает список до заметок с этим тегом, отсортированных по заголовку (без учёта регистра).
5. Дописывание `+<второй_тег><пробел>` — AND-сужение; дописывание `-<тег><пробел>` — исключение. Незавершённый последний токен (без trailing space) не применяется.
6. Wildcard-токены работают: `*` — любая последовательность (`prefix*`, `*suffix`, `*middle*`); `?` — ровно один символ (`t?g`). Точные токены и wildcard-токены могут смешиваться в одном prompt'е (`go mdx* -*draft*`).
7. Ввод тега или паттерна, которым ничего не соответствует, оставляет picker открытым с пустым списком — без уведомлений.
8. Выбор заметки переключает текущий буфер на эту заметку (`vim.cmd.edit`). Esc закрывает picker без побочных правок.
9. На отсутствие подключённого `mdx`-LSP-клиента команда `:MdxTagSearch` отвечает уведомлением-ошибкой, picker не открывается.
10. `setup({ keymaps = { tag_search = false } })` отключает keymap, не задевая команду `:MdxTagSearch` и работу `<leader>mf`/`<leader>ms`/`<leader>mi`.
11. README содержит инструкцию по подключению, грамматику prompt'а (включая wildcards) и пример сценария.

После выполнения этих пунктов M4 закрыт; цикл «найти заметки по тегам → открыть нужную» работает целиком в neovim, на источнике из БД.
