# M4 — поиск по тегам

## Цель

После M4 в markdown-буфере нажатие `<leader>mt` (или выполнение `:MdxTagSearch`) открывает Telescope-picker со списком всех тегов, известных системе, отсортированных по убыванию количества заметок. Выбор тега открывает второй Telescope-picker — со списком заметок, у которых есть этот тег. Выбор заметки открывает её в текущем буфере (как `:MdxFollow`).

После M4 пара «тег → заметки тега → открытие» закрыта целиком в neovim, поверх **всей файловой системы**, и переиспользует индекс тегов, который наполняет M0 (`mdx scan`) и поддерживает M1 (LSP upsert на open/save). Транспортный слой `mdx/*` расширяется двумя кастомными методами по той же схеме, что введена в M3.

## Что входит и что не входит

### Входит

- Кастомный LSP-метод `mdx/listTags` на стороне Go-сервера: возвращает массив записей `{ tag, count }` из таблицы `tags`, сгруппированных по `tag` и отсортированных по `count` убыванию (при равных `count` — по `tag` без учёта регистра).
- Кастомный LSP-метод `mdx/notesByTag` на стороне Go-сервера: принимает параметр `{ tag: string }`, возвращает массив записей `{ path, title }` для заметок с этим тегом, отсортированный по `title` (формат и сортировка совпадают с `mdx/listNotes`).
- Функции `db.ListTags` и `db.NotesByTag` в Go-слое БД: подготовленные SQL-запросы по таблицам `tags` и `notes`.
- Lua-функции `tags.list()` и `tags.notes_by(tag)` в `lua/mdx/tags.lua`: синхронные JSON-RPC-вызовы к подключённому `mdx`-LSP-клиенту через `vim.lsp.buf_request_sync`-эквивалент, возвращающие массив записей либо `nil` при ошибке.
- Telescope-picker'ы в `lua/mdx/picker.lua`: `picker.tags(opts, on_select)` — источник `tags.list()`, display — `#tag (N)`; `picker.notes_by_tag(tag, opts, on_select)` — источник `tags.notes_by(tag)`, display — `Заголовок (относительный/путь)`, как в M3.
- Оркестратор `M.tag_search()` в `lua/mdx/init.lua`: открывает picker тегов, по выбору открывает picker заметок выбранного тега, по выбору открывает заметку через `vim.cmd.edit` (с экранированием пути через `vim.fn.fnameescape`).
- Команда `:MdxTagSearch`, keymap `<leader>mt` (default), регистрация в `setup()` и `ftplugin/markdown.lua`.
- Расширение `defaults.keymaps` полем `tag_search = "<leader>mt"`.
- Go-тесты на `db.ListTags`, `db.NotesByTag` и оба LSP-метода.
- Lua-тесты: smoke на `tag_search` через подмену `picker.tags`/`picker.notes_by_tag`, проверяющие, что выбранная заметка открывается в текущем буфере.

### Не входит (отложено)

- Открытие заметки выбранного тега в split — кандидат на M4.5 (по аналогии с парой `MdxFollow`/`MdxFollowSplit`).
- Поиск по нескольким тегам одновременно (логическое И/ИЛИ).
- Поиск заметок по тегу под курсором без открытия picker'а тегов — самостоятельный сюжет, кандидат на M4.5.
- Иерархия тегов: тег `a/b/c` хранится и отображается как одна строка, не разворачивается в `a`, `a/b`, `a/b/c`.
- LSP completion для тегов при наборе `#` или внутри frontmatter `tags:` — это самостоятельный сюжет.
- Conceal или подсветка тегов в буфере (treesitter-query/syntax) — не входит в M4.
- Создание новой заметки с предзаданным тегом из picker'а.
- Альтернативные picker-движки (`fzf-lua`, `snacks.picker`, `vim.ui.select`) — поддерживается только Telescope.
- Реализация поверх стандартного `workspace/symbol`. Используется кастомный RPC `mdx/*` (см. «Технические решения»).

## Технические решения

| Вопрос | Решение |
|---|---|
| Источник тегов для picker'а | Кастомный LSP-метод `mdx/listTags` к серверу `mdx`. Метод не принимает параметров, возвращает массив `{ tag: string, count: int }`. Сортировка по `count DESC, tag COLLATE NOCASE` — на стороне Go. |
| Источник заметок выбранного тега | Кастомный LSP-метод `mdx/notesByTag`. Параметр — `{ tag: string }`. Возвращает массив `{ path: string, title: string }` — тот же формат, что и `mdx/listNotes`. Сортировка по `title COLLATE NOCASE, path` — на стороне Go. |
| Имена методов | `mdx/listTags`, `mdx/notesByTag`. Префикс `mdx/` — кастомный namespace, отделяет от стандартных методов. CamelCase — соответствует общей конвенции LSP и принятой в M3. |
| Чем `mdx/listTags` лучше `workspace/symbol` | `workspace/symbol` рассчитан на символы кода с обязательным `Location`. Для тегов потребовалось бы либо синтезировать фиктивный `Location` (тег ↦ первая заметка), либо отдавать на каждый тег по записи на каждое вхождение. Кастомный метод — естественная форма «выдай агрегат по таблице `tags`», унифицирует с `mdx/listNotes` и оставляет место под `count`. |
| Транспорт | Стандартный JSON-RPC через stdio LSP-канал. Lua пользуется встроенным LSP-клиентом neovim. |
| Sync vs async LSP-запрос | Sync с `timeout_ms = 2000` на каждый из двух запросов. Picker не открывается до получения ответа. БД на старте маленькая, асинхронность с прелодером — преждевременная оптимизация (та же позиция, что в M3). |
| Регистрация двух кастомных методов в `mdxHandler` | В `lsp/internal/lsp/server.go` к существующей `switch ctx.Method` добавляются две ветви — `case "mdx/listTags"` и `case "mdx/notesByTag"`. Для второго метода требуется распарсить `ctx.Params` в `struct { Tag string \`json:"tag"\` }` (через `json.Unmarshal`); первый — без параметров. |
| Форма Go-структур | `type TagEntry struct { Tag string \`json:"tag"\`; Count int \`json:"count"\` }` — для `mdx/listTags`. Для `mdx/notesByTag` переиспользуется `db.NoteEntry` из M3. |
| `tag` как ключ запроса | Чувствительность к регистру: точное совпадение (`WHERE tag = ?`). Парсер тегов M0 не нормализует регистр — теги хранятся как написаны. |
| Иерархия | Не вводится. Тег `proj/mdx` — обычная строка. |
| Источник истины для тегов | Таблица `tags(path, tag)`, наполняется на M0/M1. M4 не пересчитывает теги — полагается на содержимое таблицы. |
| Сценарий выбора | Двух-фазный: picker тегов → `<CR>` → picker заметок тега → `<CR>` → открытие заметки `vim.cmd.edit`. Esc на любой стадии прерывает; на этом конец, во внешний picker возврата нет. |
| Открытие заметки | `vim.cmd.edit` (как `M.follow`), не split. Для split — отдельная команда (отложено в Не входит). |
| Default keymap | `<leader>mt` в markdown-буферах. Регистрируется в `ftplugin/markdown.lua` на основе `M.config.keymaps.tag_search`. |
| Скоуп активации команды | `:MdxTagSearch` доступна везде после `setup()`, но требует подключённого `mdx`-LSP-клиента. Если клиент не найден — `vim.notify("mdx: LSP client not attached", ERROR)`, picker не открывается (та же модель, что у `:MdxInsertLink`). |
| Поведение при пустых ответах | `mdx/listTags` вернул пустой массив → `vim.notify("mdx: no tags in index", INFO)`, picker не открывается. `mdx/notesByTag` вернул пустой массив → `vim.notify("mdx: no notes for tag #" .. tag, INFO)`, второй picker не открывается, к picker'у тегов возврата нет. |
| Picker engine | Telescope. Зависимость уже объявлена в lazy-spec из M3. |
| Дублирование кода обёрток LSP | `tags.list()` и `tags.notes_by(tag)` повторяют структуру `notes.list()` (получение клиента → sync request → разбор `err`/`result`). Допускается осознанно: на M4 трёх вызовов недостаточно для выделения `lua/mdx/lsp.lua`-хелпера. Если на M5/M7 появятся новые методы — рефакторинг отдельным шагом. |
| Тестовая стратегия Go | `db.ListTags`/`db.NotesByTag` тестируются на in-memory SQLite с предзаполненными `notes` и `tags`. LSP-методы тестируются через расширение `lsp/internal/lsp/e2e_test.go` (или `handlers_test.go`) — отправка JSON-RPC-запроса серверу, проверка ответа. |
| Тестовая стратегия Lua | `tag_search` smoke-тестируется через подмену `picker.tags` и `picker.notes_by_tag`: подмены немедленно вызывают `on_select` с заранее заданными значениями, итог — открытие заданного буфера. Сами `tags.lua` (обёртки над LSP) не тестируются — мокировать LSP-клиент в headless-плёнэри неоправданно сложно (та же позиция, что в M3 для `notes.lua`). Telescope-UI не тестируется. |

### Шаг 2. LSP-методы `mdx/listTags` и `mdx/notesByTag`

Реализовать выборки из БД и серверные handler'ы.

#### Состав `lsp/internal/db/queries.go`

`ListTags(db *sql.DB)` выполняет запрос:

```sql
SELECT tag, COUNT(*) AS cnt
FROM tags
GROUP BY tag
ORDER BY cnt DESC, tag COLLATE NOCASE
```

Возвращает массив `TagEntry`. При пустой таблице — пустой массив, не `nil`.

`NotesByTag(db *sql.DB, tag string)` выполняет запрос:

```sql
SELECT n.path, COALESCE(n.title, '') AS title
FROM notes n
JOIN tags t ON t.path = n.path
WHERE t.tag = ?
ORDER BY (COALESCE(n.title, '') = '') ASC,
         n.title COLLATE NOCASE,
         n.path
```

Возвращает массив `NoteEntry`. При отсутствии заметок с таким тегом — пустой массив, не `nil`. Замена пустого `title` на basename без `.md` происходит на стороне handler'а (см. ниже), не в SQL — по аналогии с `onListNotes` из M3.

#### Состав `lsp/internal/lsp/handlers.go`

`onListTags(ctx)`:

1. Берёт `s.conn`.
2. Вызывает `db.ListTags(s.conn)`.
3. Возвращает результат либо ошибку, как есть.

`onNotesByTag(ctx, tag)`:

1. Берёт `s.conn`.
2. Вызывает `db.NotesByTag(s.conn, tag)`.
3. Подставляет basename без `.md` в `entries[i].Title`, если он пустой (точно так же, как в `onListNotes` из M3 step 7).
4. Пересортировывает результат после fallback'а: case-insensitive по `Title`, потом по `Path`.
5. Возвращает результат.

Регистрация методов в `mdxHandler.Handle` уже выполнена в Шаге 1.

#### Тесты

`lsp/internal/db/queries_test.go` — добавить кейсы:

- `ListTags`: пустая таблица → пустой массив; несколько тегов с разным `count` → отсортированы по `count DESC`; при равных `count` — по `tag` без учёта регистра.
- `NotesByTag`: тег без заметок → пустой массив; тег с одной/несколькими заметками → отсортированы по `title`; запись с пустым `title` оказывается в конце (на уровне SQL — fallback ставит handler).

`lsp/internal/lsp/handlers_test.go` или `e2e_test.go` — JSON-RPC-вызовы:

- `mdx/listTags` — структура и порядок ответа.
- `mdx/notesByTag` с заданным `{ tag: "alpha" }` — структура и порядок ответа; проверка fallback'а пустого `title` на basename.

**Готовность:** все Go-тесты зелёные. Ручная проверка: ручной JSON-RPC-запрос к `mdx-lsp` (`mdx/listTags` и `mdx/notesByTag`) возвращает корректный JSON.

### Шаг 3. Lua-обёртки `tags.list()` и `tags.notes_by(tag)`

Связать Lua с серверными методами.

#### Состав `lua/mdx/tags.lua`

```lua
function M.list()
function M.notes_by(tag)
```

Поведение `M.list()` (структурно повторяет `notes.list()` из M3):

1. Получает первый клиент `mdx`-LSP через `vim.lsp.get_clients({ name = "mdx" })`. Если клиента нет — `vim.notify("mdx: LSP client not attached", ERROR)` и `nil`.
2. `client:request_sync("mdx/listTags", vim.empty_dict(), 2000, 0)`.
3. На отсутствие ответа (timeout) — `vim.notify("mdx: listTags timed out", ERROR)` и `nil`.
4. На `response.err ~= nil` — `vim.notify("mdx: listTags error: " .. msg, ERROR)` и `nil`.
5. Возвращает `response.result` (массив `{ tag, count }`).

Поведение `M.notes_by(tag)` — аналогично, метод `mdx/notesByTag`, параметр `{ tag = tag }` (обычная Lua-таблица, JSON-RPC закодирует как объект).

**Готовность:** в открытом markdown-буфере `:lua print(vim.inspect(require('mdx.tags').list()))` выводит массив тегов. `:lua print(vim.inspect(require('mdx.tags').notes_by('mdx')))` выводит массив заметок (если такой тег есть). На отсутствие LSP-клиента — уведомление-ошибка и `nil`.

### Шаг 4. Picker по тегам

Реализовать `picker.tags(opts, on_select)`.

#### Состав `lua/mdx/picker.lua`

Добавить функцию:

```lua
function M.tags(opts, on_select)
```

Поведение:

1. `pcall(require, "telescope.pickers")` — если `false`, `vim.notify("mdx: telescope.nvim is required", ERROR)` и выход.
2. `entries = require("mdx.tags").list()`. Если `nil` — выход (notify уже сделан внутри `tags.list`). Если пусто — `vim.notify("mdx: no tags in index", INFO)` и выход.
3. Открывает telescope: `pickers.new`, `finders.new_table` с `entry_maker`:

   ```lua
   entry_maker = function(entry)
       return {
           value = entry,
           display = string.format("#%s (%d)", entry.tag, entry.count),
           ordinal = entry.tag,
       }
   end,
   ```

4. На `<CR>`: `on_select(selected.value.tag)`, `actions.close(prompt_bufnr)`. Передаётся именно строка-тег, не вся структура — это упрощает контракт `on_select` в `tag_search`.
5. `previewer` не задаётся (для тегов не нужен).
6. `prompt_title = "mdx: select tag"`.

**Готовность:** ручная проверка через временный вызов `:lua require('mdx.picker').tags({}, function(tag) print(tag) end)` — открывается picker тегов с `#tag (N)`-форматом, выбор печатает строку тега в `:messages`.

### Шаг 5. Picker заметок выбранного тега и оркестрация `M.tag_search()`

Реализовать `picker.notes_by_tag(tag, opts, on_select)` и связать обе фазы в `M.tag_search`.

#### Состав `lua/mdx/picker.lua`

Добавить функцию:

```lua
function M.notes_by_tag(tag, opts, on_select)
```

Поведение — структурно повторяет `picker.open` из M3, но источник заметок — `require("mdx.tags").notes_by(tag)`, заголовок picker'а — `string.format("mdx: notes #%s", tag)`, пустой ответ — `vim.notify("mdx: no notes for tag #" .. tag, INFO)`. Display, ordinal, previewer — как в `picker.open` (`require("mdx.insert").to_display_path(entry.path)`, `string.format("%s (%s)", entry.title, display_path)`, `entry.title .. " " .. entry.path`).

#### Состав `lua/mdx/init.lua`

```lua
function M.tag_search()
    require("mdx.picker").tags(M.config, function(tag)
        require("mdx.picker").notes_by_tag(tag, M.config, function(entry)
            vim.cmd.edit(vim.fn.fnameescape(entry.path))
        end)
    end)
end
```

#### Тесты

`tests/tag_search_spec.lua` — smoke-тест:

1. Подменить `require("mdx.picker").tags`: вместо реального telescope немедленно вызвать `on_select("alpha")`.
2. Подменить `require("mdx.picker").notes_by_tag`: проверить, что вызвана с `tag = "alpha"`, и немедленно вызвать `on_select({ path = "/tmp/foo.md", title = "Foo" })`.
3. Вызвать `require("mdx").tag_search()`.
4. Проверить, что `vim.api.nvim_buf_get_name(0)` после вызова — `/tmp/foo.md` (буфер переключился через `vim.cmd.edit`).

Подмены восстанавливаются в `after_each` через сохранение оригиналов (по аналогии с insert-смок-тестом из M3 step 5).

**Готовность:** Lua-тесты зелёные. Ручная проверка: в открытом markdown-буфере `:MdxTagSearch` открывает picker тегов; выбор тега открывает picker заметок этого тега; выбор заметки переключает текущий буфер на эту заметку.

### Шаг 6. Конфигурация и keymap

Расширить `setup(opts)` для управления keymap'ом.

#### Состав `lua/mdx/init.lua`

В `defaults` обновить:

```lua
keymaps = {
    follow = "<leader>mf",
    follow_split = "<leader>ms",
    insert_link = "<leader>mi",
    tag_search = "<leader>mt",   -- новое
},
```

#### Состав `ftplugin/markdown.lua`

После блока для `insert_link` добавить:

```lua
if config.keymaps.tag_search then
    vim.keymap.set("n", config.keymaps.tag_search, function()
        mdx.tag_search()
    end, { buffer = true, desc = "mdx: search notes by tag" })
end
```

**Готовность:** `<leader>mt` в markdown-буфере открывает picker тегов. `setup({ keymaps = { tag_search = false } })` отключает keymap, но команда `:MdxTagSearch` остаётся работать.

### Шаг 7. Документация

Дописать `nvim/README.md`:

- Команда `:MdxTagSearch`, поведение, требование подключённого `mdx`-LSP-клиента.
- Keymap `<leader>mt` по умолчанию.
- Указание, что источник тегов — БД, наполняемая `mdx scan` и LSP'ом на open/save. Если в индексе нет ни одного тега — picker сообщит и закроется.
- Указание, что теги извлекаются из frontmatter (`tags:` строкой через запятую или списком) и из `#tag`-вхождений в теле; иерархия не разворачивается (`a/b` — один тег).
- Сценарий «открыть picker тегов → выбрать тег → выбрать заметку → буфер переключился» как пример замкнутого цикла.
- Пометка: для просмотра выбранной заметки в split — отложено (см. M4.5).

**Готовность:** README покрывает M4-функциональность так же, как покрывает M2/M3; «как искать заметки по тегу» — один экран.

## Критерии готовности M4

1. `go test ./...` в `lsp/` зелёный, включая новые тесты на `db.ListTags`, `db.NotesByTag` и оба LSP-handler'а.
2. `nvim --headless` с `PlenaryBustedDirectory nvim/tests/` зелёный, включая новый `tag_search_spec.lua`.
3. После `mdx scan ~/projects/mdx/` БД содержит теги из заметок проекта; в открытом `.md`-файле `<leader>mt` открывает picker тегов со списком этих тегов и счётчиками.
4. Список тегов в picker'е сортируется по `count` убыванию, при равных — по `tag` без учёта регистра; display — `#tag (N)`.
5. Выбор тега открывает второй picker, отсортированный по заголовку заметки (без учёта регистра); display — `Заголовок (путь)`. Выбор заметки переключает текущий буфер на эту заметку.
6. Esc на picker'е тегов или на picker'е заметок прекращает сценарий; никаких побочных правок не происходит.
7. На отсутствие подключённого `mdx`-LSP-клиента команда `:MdxTagSearch` отвечает уведомлением-ошибкой, picker не открывается.
8. На пустой индекс тегов picker не открывается, выводится `mdx: no tags in index`. На выбранный тег без заметок (теоретически невозможно) — `mdx: no notes for tag #...`.
9. `setup({ keymaps = { tag_search = false } })` отключает keymap, не задевая команду `:MdxTagSearch` и работу `<leader>mf`/`<leader>ms`/`<leader>mi`.
10. README содержит инструкцию по подключению и пример сценария.

После выполнения этих пунктов M4 закрыт; цикл «тег → заметки тега → открытие» работает целиком в neovim, на источнике из БД.
