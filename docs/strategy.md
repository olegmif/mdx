# mdx — стратегия проекта

## Что и зачем

`mdx` — система работы с markdown-заметками поверх **всей файловой системы**, без понятия «vault». Любой `.md` в любом месте на диске может ссылаться на любой другой `.md`. Цель — заменить связку Obsidian + obsidian.nvim на легковесный неинвазивный инструмент, который интегрируется в обычный workflow neovim и не требует жить в выделенном каталоге.

Проект состоит из трёх частей:

1. **Бинарь `mdx`** (Go) — LSP-сервер и CLI-подкоманда `scan` (позже — `embed`).
2. **Lua-плагин для neovim** — тонкая обёртка: keymap-ы, picker, conceal, follow link.
3. **SQLite-база** метаданных — индекс заметок, исходящих ссылок, тегов, frontmatter; позже — embedding-ов.

## Ключевые решения и почему так

### Формат ссылок: стандартный markdown `[text](path)`

Не wiki-style `[[name]]`, не id-based. Пути — абсолютные с `~` или относительные. Причина: ссылки должны оставаться валидными **без БД и без плагина** — на GitHub, в `cat`, в любом другом редакторе. БД и плагин — слой ускорения и удобства, не зависимости.

Ссылки на не-`.md` файлы (например, README в проекте) тоже разрешены — система не различает «заметка» и «прочий markdown».

### Источник истины — файловая система. БД — кэш и индекс.

БД ускоряет picker, search, rename и хранит то, чего нет в файлах (например, embeddings). Но любую запись в БД должно быть возможно восстановить полным сканом. Это даёт устойчивость к рассинхрону и к потере БД.

### Filesystem-wide, без «индексируемых корней»

Initial scan проходит весь домашний каталог (с разумными исключениями: `.git`, `node_modules`, `.venv`, `target`, `.cache` и подобные). Дополнительные пути (`/mnt/...`) — конфигурируются.

### Один бинарь, несколько ролей

`mdx` запускается:
- как **LSP-сервер** (длинно живущий, neovim к нему подключается);
- как **CLI-подкоманда** для разовых операций — `mdx scan`.

LSP и CLI работают с одной и той же БД (SQLite в WAL-режиме, чтобы не блокировать друг друга).

## Архитектура

### Бинарь `mdx`

| Подкоманда | Назначение |
|---|---|
| `mdx lsp` | Запуск LSP-сервера (вызывается neovim'ом через `lspconfig`). |
| `mdx scan [path]` | Полный обход ФС, парсинг frontmatter и ссылок, обновление БД, удаление осиротевших записей. По умолчанию — `$HOME` плюс конфигурируемые корни. |
| `mdx embed` | (позже) пересчёт embeddings для изменившихся файлов. |

### LSP-сервер

При работе neovim делает следующее:
- `initialize` — соединение с БД, проверка схемы.
- `textDocument/didOpen` — парсинг файла, upsert метаданных и исходящих ссылок в БД, проверка валидности ссылок, выдача diagnostics для битых.
- `textDocument/didSave` — то же самое (на случай внешних изменений между open и save).
- `textDocument/completion` — completion путей в контексте `](`.
- `workspace/symbol` — отдаёт теги для поиска (или используется кастомный RPC, в зависимости от того, что удобнее в Telescope).

Битые ссылки помечаются диагностикой, но не чинятся. Ремонт — на стороне пользователя обычными средствами nvim (`cw`, `:s`, `rg --files | xargs sed`).

### Lua-плагин в neovim

- Регистрация LSP через `lspconfig`.
- Команды и keymap-ы:
  - `:MdxFollow` / `<CR>` — открыть файл по ссылке под курсором в текущем буфере.
  - `:MdxFollowSplit` / `<C-w>v` или подобный — открыть в вертикальном сплите.
  - `:MdxInsertLink` — picker по существующим заметкам (Telescope, источник — SQL-запрос к БД), вставка `[title](path)` в позиции курсора.
  - `:MdxTagSearch` — picker по тегам (через LSP).
- Conceal-конфигурация: путь в ссылке скрывается, остаётся только `text` с подчёркиванием. При курсоре на строке путь автоматически раскрывается (`conceallevel=2`, `concealcursor=""`). Реализация — через treesitter-query или vim-syntax.

### БД (SQLite + WAL)

Минимальная схема:

```
notes(
  path           TEXT PRIMARY KEY,
  mtime          INTEGER,
  size           INTEGER,
  content_hash   TEXT,
  frontmatter    JSON,
  title          TEXT
)

links(
  source_path    TEXT,
  target_path    TEXT,
  line           INTEGER,
  col            INTEGER
)

tags(
  path           TEXT,
  tag            TEXT
)

embeddings(           -- позже
  path           TEXT,
  content_hash   TEXT,
  vector         BLOB
)
```

`content_hash` хранится сразу, даже до появления embeddings — нужен для будущей инвалидации векторов.

## Надёжность и её границы

mdx — индексатор и навигатор, не валидатор графа во времени. Поддержка целостности ссылок при перемещениях и удалениях файлов **не входит в скоуп**: битые ссылки подсвечиваются диагностикой, ремонт — на пользователе обычными средствами nvim.

| Сценарий | Поведение |
|---|---|
| Правка `.md` в neovim | LSP обновляет БД на лету. |
| Внешняя правка содержимого вне neovim | БД устаревает до следующего `didOpen`/`didSave` или `mdx scan`. |
| Перемещение, удаление, реорганизация любым способом | Ссылки на затронутые пути становятся битыми и подсвечиваются диагностикой. Чинятся вручную (`cw`, `:s`, `rg --files \| xargs sed`). После — `mdx scan` для обновления индекса. |
| Sync между машинами | Каждая машина — своя БД. После большого sync — `mdx scan` на принимающей стороне. |

## План реализации (milestone-ы)

Каждый milestone оставляет систему в работоспособном состоянии. Можно использовать после любого шага.

### M0 — каркас

- Go-проект, SQLite с WAL, схема и миграции.
- CLI: `mdx scan <path>` обходит каталог, парсит frontmatter и ссылки, заполняет БД.
- Базовая обработка excludes (`.git`, `node_modules`, и т.п.).

**Результат:** разовый скан работает, БД заполняется.

### M1 — LSP skeleton

- LSP-сервер: `initialize`, `shutdown`, `textDocument/didOpen`, `textDocument/didSave`.
- На open/save — upsert метаданных и ссылок в БД, валидация ссылок, diagnostic для битых.

**Результат:** LSP цепляется в neovim, БД обновляется при работе.

### M2 — neovim-плагин: навигация

- Lua-плагин в репозитории `mdx` (или отдельный — обсудим).
- `MdxFollow`, `MdxFollowSplit`: парсинг ссылки под курсором, разворачивание `~`, открытие.
- Conceal путей с подчёркиванием title.

**Результат:** базовая навигация по ссылкам работает.

### M3 — picker и вставка ссылок

- Telescope-extension: источник — SQL-запрос к БД (все известные заметки).
- `MdxInsertLink`: после выбора в picker вставляет `[title](path)` в позицию курсора. Путь — `~/...` если файл в `$HOME`, иначе абсолютный (правило конфигурируемо).

**Результат:** ссылки можно вставлять, не копируя путь руками.

### M4 — поиск по тегам

- Парсер тегов: `tags:` в frontmatter и `#tag` в теле.
- Telescope picker по тегам через LSP `workspace/symbol` или кастомный RPC.

**Результат:** навигация по тегам.

### M5 — embeddings и semantic search (отложено)

- Подкоманда `mdx embed`: вычисление векторов для изменившихся файлов через выбранный LLM-провайдер.
- `sqlite-vec` для хранения и поиска.
- LSP/CLI команда semantic search.

**Результат:** «найди заметки, похожие на эту», «найди заметку про X» без точных ключевых слов.

## Открытые вопросы (на потом)

- **Шаблоны новых заметок** — решается стандартными nvim-средствами (luasnip / mini.snippets), не должно быть частью `mdx`.
- **Sync между машинами** — за пределами текущего скоупа. Долгосрочное направление: внешний сервер с метаданными, который перебрасывает файлы между ПК по требованию, не храня их у себя.
- **Активная поддержка целостности графа** — отслеживание перемещений, ремонт битых ссылок через code actions, `mdx mv`, демон с fsnotify. Намеренно вне скоупа на старте: мешает редко, лечится обычным `:s` / `rg --files | xargs sed`. Возвращается к рассмотрению по реальному опыту эксплуатации.

## Технические настройки

- **Язык бинаря:** Go.
- **БД:** SQLite в WAL-режиме. Драйвер — на старте `modernc.org/sqlite` (pure-Go, проще distribution); при упоре в производительность — `mattn/go-sqlite3` (cgo).
- **LSP-фреймворк:** `glsp` или ручная реализация поверх `jsonrpc2` — определится в M1.
- **Парсинг markdown:** `goldmark` или собственный лёгкий парсер ссылок и frontmatter (вероятно достаточно для наших нужд).
- **Picker:** Telescope.
- **Репозиторий:** отдельный на GitHub. Lua-плагин — либо в том же репо в подкаталоге, либо отдельно (решится в M2).
- **Рабочий каталог:** `/mnt/data/projects/mdx`.
