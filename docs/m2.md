# M2 — neovim-плагин: навигация

## Цель

Появляется первый собственный neovim-плагин в составе проекта. После M2 при курсоре на markdown-ссылке вида `[text](path)` нажатие настроенного keymap'а открывает целевой файл — в текущем окне или в вертикальном сплите. Путь скрывается через conceal: видна только часть `text`, подчёркнутая, как гиперссылка; при заходе курсора на строку путь разворачивается.

После M2 базовый цикл «увидел ссылку — перешёл по ней» работает в neovim без копирования путей руками. На этот фундамент M3 навешивает picker для **создания** ссылок, M4 — поиск по тегам, M5 — `:MdxRename`.

## Что входит и что не входит

### Входит

- Lua-плагин в подкаталоге `nvim/` репозитория `mdx`.
- Парсер ссылки `[text](target)` под курсором: определяется ссылка, в чьих пределах находится курсор, извлекаются `text`, `target` и позиция.
- Резолвер `target` в абсолютный путь файловой системы: разворачивание `~`, разрешение относительно каталога текущего буфера, нормализация.
- Команды `:MdxFollow` и `:MdxFollowSplit`, открывающие целевой файл соответственно в текущем окне (`:edit`) или в вертикальном сплите (`:vsplit`).
- Conceal-конфигурация через vim syntax, скрывающая path-часть ссылки и подчёркивающая `text`. Активируется только в `markdown`-буферах.
- Функция `require('mdx').setup(opts)` для пользовательской конфигурации: keymaps, поведение open-команд, включение/выключение conceal.
- Plugin-spec для lazy.nvim с конфигурацией по умолчанию.
- Юнит-тесты на парсер и резолвер через `plenary.nvim`.

### Не входит (отложено)

- Treesitter-based парсер ссылок. На M2 — вим-regex; миграция на treesitter возможна позже без изменения публичного API плагина.
- Picker по существующим заметкам и команда `:MdxInsertLink` — это M3.
- Поиск по тегам и команда `:MdxTagSearch` — это M4.
- `:MdxRename` — это M5.
- Подсветка содержимого ссылки в виде иконки или префикса перед `text`.
- Поддержка `#section`-якорей внутри целевого файла (открытие файла со скроллом до раздела).
- Image-embed (`![alt](path)`) на M2 трактуется так же, как обычная ссылка; preview-функциональность не предусмотрена.

## Технические решения, которые надо зафиксировать перед стартом

| Вопрос | Решение |
|---|---|
| Расположение плагина | Подкаталог `nvim/` в репозитории `mdx`. Позже, при необходимости публикации, можно вынести в отдельный репозиторий — публичный API плагина и расположение файлов под `lua/mdx/` это переживут. |
| Минимальная версия neovim | `0.11+`. Обоснование: проект уже использует `vim.lsp.config` API из этой ветки, единая планка снимает разнобой. |
| Парсер ссылок | Vim regex через `vim.fn.matchstrpos`. Полностью покрывает синтаксис `[text](target)` плюс опциональный `"title"` после target'а; не требует установленного treesitter-парсера. |
| Conceal | Vim syntax (`syn match`) c флагом `conceal`. Альтернатива через treesitter возможна, но добавляет зависимость и не даёт ощутимых преимуществ. |
| Скоуп активации | Только `filetype=markdown`. Реализуется через `ftplugin/markdown.lua` — стандартный механизм neovim для filetype-локальных настроек. |
| Тестовый фреймворк | `nvim-lua/plenary.nvim` (`PlenaryBustedDirectory`). Это де-факто стандарт для тестирования lua-плагинов. |
| Default keymaps | Активны в markdown-буферах: `<CR>` → `MdxFollow`, `<C-w>v` → `MdxFollowSplit`. Пользователь может переопределить или отключить через `setup({ keymaps = ... })`. |
| Fallback при отсутствии ссылки | `MdxFollow()` возвращает `false`. Стандартный keymap-обёртка вызывает поведение по умолчанию (`<CR>` → следующая строка) при `false`, чтобы не ломать привычное UX. |
| Не-`file` ссылки (URL) | Пропускаются с уведомлением через `vim.notify("mdx: external URL, ignored")`. Открытие в браузере — отдельная фича вне M2. |

## Шаги выполнения

Шаги упорядочены так, чтобы после каждого можно было запустить осмысленную проверку — либо тест, либо ручной сценарий в neovim.

### Шаг 1. Скаффолдинг плагина

Создать каталог `nvim/` с минимальной структурой и заглушкой публичного API.

#### Состав каталога `nvim/`

`lua/mdx/init.lua` объявляет публичные функции:

```lua
function M.setup(opts)
function M.follow()
function M.follow_split()
```

`setup(opts)` принимает таблицу пользовательских настроек, мерджит с дефолтами и сохраняет в module-локальной переменной. На этом шаге функции возвращают пустые значения; цель — проверить, что плагин подгружается.

`ftplugin/markdown.lua` оставляется пустым — содержательно заполняется в Шаге 5.

#### Установка через lazy.nvim

В `nvim/README.md` приводится spec для локальной разработки:

```lua
{
    dir = "/home/oleg/projects/mdx/nvim",
    name = "mdx",
    ft = "markdown",
    config = function()
        require("mdx").setup({})
    end,
}
```

**Готовность:** после `:Lazy reload mdx` команда `:lua print(require('mdx'))` печатает таблицу с тремя функциями, ошибок в `:messages` нет.

### Шаг 2. Парсер ссылки под курсором

Реализовать функцию, извлекающую markdown-ссылку, в чьих пределах находится курсор.

#### Состав `lua/mdx/link.lua`

```lua
function M.under_cursor() -- returns table or nil
```

Получает текущую строку через `vim.api.nvim_get_current_line()` и колонку курсора через `vim.api.nvim_win_get_cursor(0)`. Использует `vim.fn.matchstrpos(line, pattern)` в цикле для перебора всех вхождений ссылки на строке. Для каждого совпадения проверяет, попадает ли позиция курсора в диапазон `[start, end)`. Возвращает таблицу:

```lua
{
    text   = "...",   -- что в [...] 
    target = "...",   -- что в (...)
    start  = N,       -- 0-based колонка начала "["
    finish = N,       -- 0-based колонка после ")"
}
```

или `nil`, если курсор не на ссылке.

Регулярное выражение (vim regex): `\[\([^]]*\)\](\([^) \t]\+\)\%(\s\+"[^"]*"\)\=)`.

#### Тесты

`tests/link_spec.lua` использует `vim.api.nvim_buf_set_lines` для подготовки буфера и `vim.api.nvim_win_set_cursor` для позиционирования курсора. Покрывает: одна ссылка на строке, несколько ссылок на строке (курсор внутри второй), курсор не на ссылке, ссылка с title.

**Готовность:** `nvim --headless -c "PlenaryBustedDirectory tests/" -c "qa!"` выполняется без ошибок.

### Шаг 3. Резолвер пути

Реализовать функцию, превращающую `target` ссылки в абсолютный путь файловой системы.

#### Состав `lua/mdx/resolve.lua`

```lua
function M.target_to_path(target, source_dir) -- returns string or nil
```

Поведение:

1. Если `target` начинается с одной из URL-схем (`http://`, `https://`, `mailto:`, `ftp://`, `tel:`, `#`) — возвращает `nil`. Эти случаи отдельно обрабатываются вызывающим кодом.
2. Если `target` начинается с `~/` или равен `~` — разворачивает через `vim.fn.expand("~")`, конкатенирует остаток.
3. Если `target` начинается с `/` — возвращает как есть после нормализации `vim.fs.normalize`.
4. Иначе — резолвит относительно `source_dir`: `vim.fs.normalize(source_dir .. "/" .. target)`.

`source_dir` передаётся вызывающим кодом и обычно равен `vim.fs.dirname(vim.api.nvim_buf_get_name(0))`.

#### Тесты

`tests/resolve_spec.lua` — табличный тест на пары `(target, source_dir, expected)` для случаев: относительный путь с `..`, абсолютный путь, путь с `~`, URL-схема (ожидается `nil`), якорь `#section` (ожидается `nil`).

**Готовность:** все табличные кейсы зелёные.

### Шаг 4. Команды `MdxFollow` и `MdxFollowSplit`

Связать парсер и резолвер в единый сценарий открытия файла.

#### Состав `lua/mdx/init.lua`

```lua
function M.follow()        -- returns boolean
function M.follow_split()  -- returns boolean
```

Поведение:

1. Вызывают `link.under_cursor()`. Если `nil` — возвращают `false`.
2. Вызывают `resolve.target_to_path(link.target, source_dir)`. Если `nil` — выводят `vim.notify("mdx: external URL, ignored", vim.log.levels.INFO)` и возвращают `true` (сценарий обработан, пусть и без открытия).
3. Открывают файл соответствующей командой:
    - `M.follow()` — `vim.cmd.edit(vim.fn.fnameescape(path))`.
    - `M.follow_split()` — `vim.cmd.vsplit(vim.fn.fnameescape(path))`.
4. Возвращают `true`.

Возвращаемое значение нужно для keymap-обёртки, которая откатывается на стандартное поведение `<CR>` при отсутствии ссылки под курсором (см. Шаг 6).

**Готовность:** при курсоре на `[Test](./test.md)` в открытом `.md` файле выполнение `:lua require('mdx').follow()` открывает `test.md` в текущем окне; `:lua require('mdx').follow_split()` открывает в вертикальном сплите.

### Шаг 5. Conceal

Скрыть path-часть ссылки и подчеркнуть `text` так, чтобы ссылка визуально читалась как гиперссылка.

#### Состав `ftplugin/markdown.lua`

При загрузке буфера выполняется:

```lua
vim.opt_local.conceallevel    = 2
vim.opt_local.concealcursor   = ""
```

`conceallevel=2` включает conceal с замещением; `concealcursor=""` гарантирует, что при заходе курсора на строку весь текст разворачивается полностью.

#### Состав `lua/mdx/conceal.lua`

```lua
function M.attach(bufnr)
```

Через `vim.api.nvim_buf_call(bufnr, ...)` применяет к буферу два syntax-правила:

```vim
syn match mdxLinkText "\[\zs[^]]*\ze\](\([^) \t]\+\)\%(\s\+"[^"]*"\)\=)" containedin=ALL
syn match mdxLinkPath "\[[^]]*\]\zs(\([^) \t]\+\)\%(\s\+"[^"]*"\)\=)\ze" conceal containedin=ALL
hi default link mdxLinkText Underlined
```

Первое правило выделяет текст ссылки и подсвечивает его как `Underlined`; второе скрывает path-часть.

`ftplugin/markdown.lua` вызывает `require('mdx.conceal').attach(0)` для текущего буфера.

**Готовность:** при открытии `.md` файла со ссылкой `[Test](./test.md)` отображается только подчёркнутый `Test`; при перемещении курсора на строку видна полная ссылка.

### Шаг 6. Конфигурация через `setup()`

Реализовать пользовательскую конфигурацию: keymaps, переключение conceal, переопределение open-команд.

#### Состав `lua/mdx/init.lua`

```lua
local defaults = {
    keymaps = {
        follow       = "<CR>",
        follow_split = "<C-w>v",
    },
    conceal = true,
}

function M.setup(opts)
```

`setup(opts)` глубоко мерджит `opts` с `defaults` и сохраняет результат в `M.config`. Никаких побочных эффектов на уровне глобального состояния не выполняет; все настройки применяются per-buffer в `ftplugin/markdown.lua`.

#### Изменения в `ftplugin/markdown.lua`

После применения conceal:

1. Если `M.config.conceal == false` — пропустить `conceal.attach`.
2. Для каждого keymap из `M.config.keymaps`, если значение не `false`, навесить bufer-локальный keymap. Маппинг `follow` использует обёртку:

```lua
vim.keymap.set("n", M.config.keymaps.follow, function()
    if not require("mdx").follow() then
        vim.api.nvim_feedkeys(
            vim.api.nvim_replace_termcodes(M.config.keymaps.follow, true, false, true),
            "n", false
        )
    end
end, { buffer = true })
```

Таким образом нажатие `<CR>` без ссылки под курсором даёт стандартное поведение «следующая строка».

**Готовность:** в обычной строке `<CR>` ведёт себя как раньше; на строке со ссылкой — открывает целевой файл. `setup({ conceal = false })` отключает скрытие путей. `setup({ keymaps = { follow_split = false } })` отключает соответствующий keymap.

## Критерии готовности M2

1. `:Lazy reload mdx` подгружает плагин без ошибок в `:messages`.
2. `nvim --headless -c "PlenaryBustedDirectory nvim/tests/" -c "qa!"` зелёный.
3. В открытом `.md` файле курсор на `[Test](./test.md)`, `<CR>` открывает `test.md` в текущем окне; `<C-w>v` — в вертикальном сплите.
4. На строке без ссылки `<CR>` ведёт себя как стандартный keymap (переход на следующую строку).
5. Path-часть ссылки скрыта, виден подчёркнутый `text`. При заходе курсора на строку отображается вся ссылка целиком.
6. `setup({ conceal = false })` отключает скрытие; `setup({ keymaps = { follow = false } })` отключает соответствующий keymap.
7. README содержит минимальный пример подключения плагина.

## Что нужно подтвердить, прежде чем стартовать

1. **Расположение плагина.** Предложено: подкаталог `nvim/` в репозитории `mdx`. Альтернатива — отдельный репозиторий, что упростит публикацию через стандартные plugin-менеджеры, но удвоит количество мест, где живёт код проекта.
2. **Default keymaps в markdown-буферах.** Предложено: `<CR>` для follow, `<C-w>v` для follow_split. Альтернатива — нейтральный префикс (`<leader>m...`), оставляющий `<CR>` без изменений, но требующий отдельного знания.
3. **Conceal-движок.** Предложено: vim syntax. Альтернатива — treesitter, что точнее парсит вложенные конструкции, но требует наличия `nvim-treesitter` с парсером `markdown`.
