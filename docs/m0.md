# M0 — каркас

## Цель

Получить работающую CLI-команду `mdx scan <path>`, которая обходит указанный каталог, находит все `.md`-файлы (с разумными исключениями), извлекает из каждого метаданные, исходящие ссылки и теги, и сохраняет всё это в SQLite-базу. После M0 у нас есть фундамент: проект на Go, БД с правильной схемой, парсер markdown-файлов и обходчик ФС. На этот фундамент в следующих milestone-ах ставится LSP-сервер и подкоманды `mv`, `gc`, `embed`.

## Что входит и что не входит

### Входит

- Go-модуль в каталоге `/mnt/data/projects/mdx/lsp`.
- SQLite-база с WAL и схемой согласно стратегии (таблицы `notes`, `links`, `tags`).
- Простой механизм миграций (версия схемы в БД, применение нужных миграций при старте).
- Подкоманда `mdx scan` с обходом ФС, парсингом и записью в БД.
- Базовый набор excludes: `.git`, `node_modules`, `.venv`, `target`, `.cache`, `dist`, `build`, `vendor`.
- Тесты на парсер и end-to-end на скан фикстурного каталога.
- Минимальный README в `lsp/` с инструкцией сборки и запуска.

### Не входит (отложено)

- LSP-сервер (M1).
- Подкоманды `mv`, `gc`, `embed`.
- Таблица `embeddings` (создадим её, когда дойдём до M8 — нет смысла плодить пустые сущности заранее).
- `.gitignore`-aware excludes (хардкод-список достаточен для старта).
- Code blocks awareness при парсинге inline-тегов (можем переловить чуть лишних `#tag` в коде — пофиксим, когда станет проблемой).
- Watcher-режим, инкрементальный скан по mtime (на M0 — всегда полный обход указанного пути).
- Прогресс-бар (только итоговая сводка).

## Технические решения, которые надо зафиксировать перед стартом

| Вопрос | Предлагаемое решение |
|---|---|
| Имя Go-модуля | `github.com/<user>/mdx/lsp` — нужно подтвердить github-аккаунт. До подтверждения — placeholder, например `mdx/lsp`. |
| Драйвер SQLite | `modernc.org/sqlite` (pure Go, без cgo). Стратегия это уже зафиксировала. |
| CLI-фреймворк | `spf13/cobra` — стандарт де-факто, упрощает добавление подкоманд в M1+. Альтернатива на M0 — голый `flag` + ручной диспатч; экономит зависимость, но придётся переписывать. **Берём cobra.** |
| Парсер markdown | На M0 — собственный лёгкий парсер на регулярках для ссылок, тегов и frontmatter. `goldmark` тяжёл и для нашей задачи избыточен. Если позже потребуется AST — мигрируем. |
| Парсер YAML frontmatter | `gopkg.in/yaml.v3`. |
| Алгоритм content_hash | SHA-256 (стандартная библиотека, скорость достаточна). |
| Расположение БД | `$XDG_DATA_HOME/mdx/mdx.db`, fallback `$HOME/.local/share/mdx/mdx.db`. Переопределяется флагом `--db` или переменной `MDX_DB`. |
| Конфигурация excludes | Хардкод-список в M0. Флаг `--exclude` для добавления — опционально. |

## Схема БД (версия 1)

```sql
CREATE TABLE schema_version (
  version INTEGER PRIMARY KEY
);

CREATE TABLE notes (
  path          TEXT PRIMARY KEY,
  inode         INTEGER,
  mtime         INTEGER NOT NULL,
  size          INTEGER NOT NULL,
  content_hash  TEXT NOT NULL,
  frontmatter   TEXT,                      -- JSON-сериализованный frontmatter
  title         TEXT
);

CREATE INDEX idx_notes_inode        ON notes(inode);
CREATE INDEX idx_notes_content_hash ON notes(content_hash);

CREATE TABLE links (
  source_path  TEXT NOT NULL,
  target_path  TEXT NOT NULL,              -- абсолютный путь, разрешённый относительно source
  raw_target   TEXT NOT NULL,              -- что реально написано в файле
  line         INTEGER NOT NULL,
  col          INTEGER NOT NULL,
  FOREIGN KEY (source_path) REFERENCES notes(path) ON DELETE CASCADE
);

CREATE INDEX idx_links_source ON links(source_path);
CREATE INDEX idx_links_target ON links(target_path);

CREATE TABLE tags (
  path  TEXT NOT NULL,
  tag   TEXT NOT NULL,
  PRIMARY KEY (path, tag),
  FOREIGN KEY (path) REFERENCES notes(path) ON DELETE CASCADE
);

CREATE INDEX idx_tags_tag ON tags(tag);
```

`raw_target` нужен в `links`, чтобы при rename правильно восстановить написание (абсолютный/относительный/`~/...`). `target_path` всегда нормализованный абсолютный — для быстрых джойнов и поиска.

## Pragma при открытии БД

```sql
PRAGMA journal_mode=WAL;
PRAGMA synchronous=NORMAL;
PRAGMA foreign_keys=ON;
PRAGMA temp_store=MEMORY;
```

## Шаги выполнения

Шаги упорядочены так, чтобы после каждого можно было запустить и увидеть осмысленный результат.

### Шаг 1. Скаффолдинг Go-проекта

- В `lsp/` сделать `go mod init <имя-модуля>`.
- Добавить `cmd/mdx/main.go` с пустым `main()`, который вызовет диспатч cobra.
- Создать пустые пакеты `internal/cli`, `internal/db`, `internal/scan`, `internal/parse`.
- `Makefile` с целями `build` (`go build -o bin/mdx ./cmd/mdx`), `test` (`go test ./...`), `clean`.
- `.gitignore` для Go (`bin/`, `*.test`, и т.п.).

**Готовность:** `make build` собирает бинарь, `./bin/mdx --help` печатает заглушку cobra.

### Шаг 2. Подключение SQLite и схема

- Реализовать `internal/db/db.go`: функция `Open(path string) (*sql.DB, error)`, открывает файл, применяет pragmas.
- `internal/db/migrations.go`: функция `Migrate(db *sql.DB)`, читает текущую версию из `schema_version`, применяет нужные SQL-файлы.
- `internal/db/schema.sql` — embed через `//go:embed` или просто строка в коде; SQL-скрипт версии 1 как выше.
- Резолвинг пути к БД: `MDX_DB` env → `--db` флаг → `$XDG_DATA_HOME/mdx/mdx.db` → `$HOME/.local/share/mdx/mdx.db`. Если каталог не существует — создать.

**Готовность:** unit-тест: открыть БД во временном файле, проверить, что миграция прошла и таблицы созданы.

### Шаг 3. Парсер frontmatter

- `internal/parse/frontmatter.go`: функция `Parse(content []byte) (frontmatter map[string]any, body []byte, err error)`.
- Распознавать `---` в начале файла, искать закрывающий `---` или `...`, парсить YAML между ними. Если frontmatter отсутствует — вернуть `nil` и весь контент как body.
- Извлекать `tags:` и `title:` для удобства, но не обязательно — это сделает caller.

**Готовность:** табличный тест на пары `(input, expected_frontmatter, expected_body)` для случаев: нет frontmatter, есть с `tags`, есть с не-yaml между маркерами (ошибка).

### Шаг 4. Парсер ссылок

- `internal/parse/links.go`: функция `Extract(body []byte) []Link`, где `Link` содержит `Text`, `RawTarget`, `Line`, `Col`.
- Регулярка: `\[([^\]\n]*)\]\(([^)\s]+)(?:\s+"[^"]*")?\)`.
- Фильтровать ссылки, которые точно не файловые: схемы `http://`, `https://`, `mailto:`, `ftp://`, `tel:`, `#` (внутри-документные якоря).
- Для оставшихся — ничего не резолвим в этом модуле, возвращаем как есть. Резолвинг в абсолютный путь — задача scan-pipeline (он знает каталог исходного файла).

**Готовность:** табличный тест на текстовые сэмплы.

### Шаг 5. Парсер тегов

- `internal/parse/tags.go`: функция `Extract(frontmatter map[string]any, body []byte) []string`.
- Из frontmatter — поле `tags` (строка-через-запятую или массив строк).
- Из body — регулярка `(?:^|[^\w&])#([\w][\w/-]*)`. Без awareness code-block-ов на M0.
- Дедуп.

**Готовность:** табличный тест.

### Шаг 6. Walker

- `internal/scan/walker.go`: функция `Walk(root string, excludes []string, fn func(path string, info fs.FileInfo) error) error`.
- На основе `filepath.WalkDir`. На каждом каталоге проверять имя против excludes — если совпадает, `fs.SkipDir`.
- Вызывать callback только для файлов с расширением `.md`.

**Готовность:** unit-тест: создать временное дерево с `.md`, не-`.md` и каталог из excludes; проверить, что callback вызвался ровно для нужных файлов.

### Шаг 7. Stat и хеш

- `internal/scan/stat.go`: функция `Stat(path string) (mtime, size, inode int64, err error)`. Для inode на linux/mac — `syscall.Stat_t.Ino`. На windows — 0 (windows вне скоупа сейчас, но не сломаем компиляцию).
- `internal/scan/hash.go`: функция `Hash(path string) (string, error)`, читает файл, считает SHA-256, возвращает hex.

**Готовность:** unit-тесты.

### Шаг 8. SQL-операции

- `internal/db/queries.go`:
  - `UpsertNote(tx, NoteRecord)`: INSERT OR REPLACE в `notes`.
  - `ReplaceLinks(tx, sourcePath string, links []LinkRecord)`: DELETE по source + bulk INSERT.
  - `ReplaceTags(tx, sourcePath string, tags []string)`: DELETE по path + bulk INSERT.
- Все три вызываются внутри одной транзакции на файл.

**Готовность:** unit-тест: вставить запись, прочитать обратно через прямой SQL.

### Шаг 9. Сборка scan-pipeline

- `internal/cli/scan.go`: функция `Run(ctx, db *sql.DB, roots []string, excludes []string) (Stats, error)`.
- Для каждого корня вызвать `Walk`. На каждый файл:
  1. Прочитать содержимое.
  2. Посчитать hash.
  3. `Stat` → mtime, size, inode.
  4. Парс frontmatter → `body`, `frontmatter map`.
  5. Парс ссылок из body. Резолв `RawTarget` в абсолютный путь относительно директории файла (с разворачиванием `~`). Для http/etc — пропуск (см. шаг 4).
  6. Парс тегов.
  7. В одной транзакции — `UpsertNote`, `ReplaceLinks`, `ReplaceTags`.
- Собирать статистику: сколько файлов отсканировано, сколько ошибок (с записью в stderr, но без аборта).

**Готовность:** ручной запуск на `testdata/fixtures` — БД заполнена ожидаемыми данными.

### Шаг 10. Подкоманда `mdx scan`

- В `cmd/mdx/main.go` подключить cobra.
- Команда `scan [path...]`: дефолт — `$HOME`. Флаги `--db`, `--exclude` (массив, добавляется к хардкоду), `--quiet`.
- В конце печатать сводку: `scanned: N, errors: M, elapsed: T`.

**Готовность:** `mdx scan ~/notes` (на тестовом каталоге) выводит сводку, БД заполнена.

### Шаг 11. End-to-end тест

- В `internal/cli/scan_test.go` или отдельном `e2e_test.go`:
  - Создать tmpdir, налить в него `testdata/fixtures`.
  - Открыть временную БД, запустить scan на tmpdir.
  - Запросить БД и сравнить с golden-результатом.

**Готовность:** `make test` зелёный.

### Шаг 12. README в `lsp/`

- Описание: что это, как собрать (`make build`), как запустить (`./bin/mdx scan`), где живёт БД.
- Один экран. Подробности — в `docs/strategy.md` и `docs/m0.md`.

## Критерии готовности M0

1. `make build` собирает бинарь без ошибок.
2. `make test` проходит зелёным.
3. `./bin/mdx scan ~/<тестовый каталог>` отрабатывает за разумное время и выдаёт сводку.
4. SQLite-файл создан в нужном месте, содержит таблицы `notes`, `links`, `tags` с заполненными записями для тестового каталога.
5. Повторный запуск scan не дублирует записи (UPSERT работает) и обновляет mtime/hash, если файл изменился.
6. README описывает запуск.

После выполнения этих пунктов M0 закрыт, переходим к M1 (LSP skeleton).
