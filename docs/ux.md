# mdx — улучшения UX

После реализации стратегии (M0–M5) сюда накапливаются ergonomic-фичи: операции, не меняющие архитектуру и не входящие в стратегический скоуп, но удобные в повседневной работе. Каждый пункт — самостоятельная фича без зависимостей; реализуется по мере необходимости.

---

## `:MdxExtract` — вынос выделения в новую заметку

Уносит выделенный в visual mode фрагмент в новый файл, на месте выделения оставляет ссылку. Открывает созданный файл в вертикальном split, чтобы пользователь мог сразу дополнить заметку (frontmatter, обратная ссылка, заголовок).

### Поведение

| Аспект | Поведение |
|---|---|
| Триггер | `:MdxExtract` (только в visual mode); keymap `<leader>me` |
| Имя файла по умолчанию | Таймштамп `YYYYMMDDHHMMSS.md` |
| Запрос имени у пользователя | По опции `prompt_filename` (default `false`). При `true` — `vim.ui.input` с дефолтом-таймштампом; пустой ввод → дефолт. |
| Каталог нового файла | Каталог текущего буфера (`expand("%:p:h")`) |
| Текст ссылки | Имя файла без `.md` |
| Формат пути в ссылке | Через ту же логику, что `:MdxInsertLink` (общий хелпер) |
| Содержимое нового файла | Ровно выделенный фрагмент, без frontmatter и без заголовков |
| После создания | Открыть новый файл в вертикальном split (`:vsplit`). Это и подтверждение успешной операции, и точка для немедленного дополнения. |
| Индексация нового файла | Автоматически через LSP `didOpen` при открытии split — поэтому picker и follow увидят заметку сразу, без `mdx scan`. |
| Конфликт имён (файл уже существует) | Ошибка, операция отменяется, исходный буфер не меняется. |

### Алгоритм

1. Считать диапазон выделения (`'<`, `'>`).
2. Извлечь текст; если пусто — сообщение, выход.
3. По опции `prompt_filename`: либо запросить имя через `vim.ui.input`, либо использовать таймштамп напрямую.
4. Собрать целевой путь от каталога текущего буфера.
5. Если файл существует — ошибка, выход (буфер не меняется).
6. Записать выделенный текст в новый файл (`vim.fn.writefile`).
7. Заменить выделение на ссылку через общий хелпер форматирования пути.
8. Открыть новый файл в вертикальном split (`:vsplit <path>`).

### Размещение в коде

- `nvim/lua/mdx/extract.lua` — реализация.
- Регистрация команды и keymap в `init.lua` рядом с прочими `:Mdx*`.
- Логика «таймштамп + опциональный prompt» — в общем хелпере `nvim/lua/mdx/filename.lua`, переиспользуется с `:MdxNew`.
- Хелпер форматирования пути ссылки выносится из `insert.lua` в общий модуль (`mdx/link_format.lua` либо расширение `resolve.lua`) и переиспользуется обоими callers.
- Опция `prompt_filename` добавляется в дефолты `setup({ ... })`.
- Тесты — `nvim/tests/extract_spec.lua` по образцу `insert_spec.lua`.

### Заметки

- Опция `prompt_filename` сформулирована широко («запрашивать имя файла при создании или извлечении») в расчёте на то, что в будущем появится `:MdxNew` (создание пустой заметки) с тем же поведением запроса. Если поведение со временем разойдётся — разделить на `prompt_on_extract` / `prompt_on_new`.
- Открытие в split, а не в текущем окне — сознательное решение: сохраняет контекст исходной заметки, чтобы можно было сравнить и дополнить.

## `:MdxNew` — создание новой заметки

Создаёт пустой файл и открывает его в текущем буфере, чтобы пользователь сразу начал писать.

### Поведение

| Аспект | Поведение |
|---|---|
| Триггер | `:MdxNew`; keymap `<leader>mn` |
| Имя файла по умолчанию | Таймштамп `YYYYMMDDHHMMSS.md` |
| Запрос имени у пользователя | По той же опции `prompt_filename`, что и у `:MdxExtract` (default `false`) |
| Каталог нового файла | Каталог текущего буфера. Если буфер без файла — `getcwd()`. |
| Содержимое нового файла | Пустое. |
| После создания | Открыть в текущем буфере (`:edit`). |
| Индексация нового файла | Автоматически через LSP `didOpen` при открытии. |
| Конфликт имён | Ошибка, операция отменяется. |
| Несохранённые изменения в текущем буфере | `:edit` падает со стандартной ошибкой nvim — пользователь сохраняет или отбрасывает сам. |

### Алгоритм

1. По опции `prompt_filename`: запросить имя через `vim.ui.input` с дефолтом-таймштампом, либо использовать таймштамп напрямую.
2. Собрать целевой путь от каталога текущего буфера (или `getcwd()`, если буфер без файла).
3. Если файл существует — ошибка, выход.
4. Создать пустой файл (`vim.fn.writefile({}, path)`).
5. Открыть в текущем буфере (`vim.cmd.edit(path)`).

### Размещение в коде

- `nvim/lua/mdx/new.lua` — реализация.
- Хелпер `nvim/lua/mdx/filename.lua` — общий с `:MdxExtract` (таймштамп + опциональный prompt).
- Регистрация команды и keymap в `init.lua`.
- Тесты — `nvim/tests/new_spec.lua`.

### Заметки

- Каталог по умолчанию — каталог текущего буфера: заметки обычно создаются в контексте того, что сейчас открыто. Для глобального «inbox» можно ввести опцию `new_note_directory` позже, если появится потребность.

## `:MdxSql` — picker по заранее заготовленным SQL-запросам

Telescope-picker, в котором показывается список SQL-запросов, лежащих файлами в каталоге настроек. Пользователь выбирает запрос — он выполняется на сервере (только SELECT/WITH), результат открывается во втором picker'е, в котором каждая строка — заметка с возможностью открыть её на Enter.

### Поведение

| Аспект | Поведение |
|---|---|
| Триггер | `:MdxSql`; keymap `<leader>mq` |
| Каталог запросов | По опции `query_dir` (default `~/.config/mdx/queries`) |
| Что в файле | Обычный `.sql` файл с одним запросом. Имя файла без `.sql` — отображаемое название. |
| Описание (опционально) | Первая строка вида `-- description: ...` подмешивается к названию в picker'е |
| Ограничение SQL | Сервер принимает только `SELECT` / `WITH` (после удаления комментариев); всё остальное отклоняется до выполнения |
| Колонки результата | Минимум `path`. Если есть `title` — используется в отображении; иначе fallback на basename файла. Прочие колонки игнорируются (показываются только path/title). |
| Каталог не существует | Сообщение, выход. |
| Запрос не вернул строк | Сообщение, выход. |
| Колонка `path` отсутствует | Ошибка, выход. |

### Пример

`~/.config/mdx/queries/active-tasks.sql`:

```sql
-- description: type=task & status=active
SELECT path, COALESCE(title, '') AS title
FROM notes
WHERE json_extract(frontmatter, '$.type')   = 'task'
  AND json_extract(frontmatter, '$.status') = 'active'
ORDER BY title COLLATE NOCASE
```

После `<leader>mq`:
1. picker «mdx: sql queries» показывает `active-tasks — type=task & status=active`.
2. Enter — запрос выполняется.
3. picker «mdx: active-tasks (N)» показывает все совпавшие заметки.
4. Enter на строке — открывается соответствующий файл.

### Алгоритм

1. По опции `query_dir` найти каталог. Если нет — сообщение, выход.
2. Прочитать все `*.sql` файлы из каталога. У каждого взять содержимое и опциональный `-- description: ...` из первой строки.
3. Открыть picker запросов (Telescope, file_previewer показывает содержимое .sql).
4. На выборе — отправить SQL на сервер через `mdx/query`.
5. Если результат пуст или нет колонки `path` — сообщение, выход.
6. Открыть picker результатов (Telescope, отображение `title (display_path)`, file_previewer показывает заметку).
7. На Enter — `vim.cmd.edit(row.path)`.

### Размещение в коде

- `nvim/lua/mdx/sql.lua` — парсинг файлов запросов, оркестрация (resolve_dir, list, run, open).
- `nvim/lua/mdx/picker.lua::sql_queries`, `::sql_results` — два Telescope-picker'а (запросов и результатов).
- `nvim/lua/mdx/query.lua` — клиент LSP-метода `mdx/query` (общий с любым future-кодом, идущим через тот же endpoint).
- На стороне сервера: `lsp/internal/db/queries.go::Query` + валидатор `validateSelectOnly`, handler `mdx/query` в `lsp/internal/lsp/server.go`.
- Тесты — `nvim/tests/sql_spec.lua`, `lsp/internal/db/queries_test.go::TestQuery*`.

### Заметки

- Никакого исполнения произвольного кода — только SELECT/WITH через серверный валидатор. Открытие чужой заметки никогда не запускает SQL само по себе.
- Запросы версионируются вместе с пользовательским dotfiles-конфигом (или просто живут в `~/.config/mdx/queries/`).
- Параметризация (запросы с placeholder'ами, заполняемыми пользователем перед выполнением) — пока не поддерживается. Если появится потребность — добавить отдельный prompt-шаг между picker'ом запросов и его выполнением.
- Schema-as-API: запросы пишутся против схемы `notes`/`links`/`tags` из `strategy.md`. При изменении схемы запросы потребуется обновлять руками.

## `:MdxQuery` и `:MdxQueryInsert` — picker по Lua-скриптам

Аналогично `:MdxSql`, но скрипты — на Lua. Это даёт обработку результата перед выводом: формат, фильтрация, аггрегация, мульти-запросы. Один и тот же скрипт обслуживает обе команды; разница только в том, что делается с результатом — показ в Telescope или вставка в буфер.

### Поведение

| Аспект | Поведение |
|---|---|
| Триггеры | `:MdxQuery` (показ); `:MdxQueryInsert` (вставка); keymap'ы `<leader>mr` и `<leader>mR` |
| Каталог скриптов | По опции `script_dir` (default `~/.config/mdx/scripts`) |
| Что в файле | `.lua` файл со скриптом. Имя файла без `.lua` — название в picker'е. |
| Описание (опционально) | Первая строка `-- description: ...` подмешивается в picker |
| Окружение скрипта | Стандартный Lua + `vim` + `mdx` (см. ниже). Sandbox'а нет — скрипты пишутся пользователем. |
| Тип результата | Любой — приводится к строке через `to_string` (см. правила) |
| `:MdxQuery` | Строка разбивается по `\n`, показывается в Telescope-picker'е (Enter ничего не делает, просто закрывает) |
| `:MdxQueryInsert` | Строка вставляется в буфер в позицию курсора через `nvim_put` |
| Пустой результат | Сообщение, выход (без открытия picker'а / без вставки) |
| Ошибка скрипта | Текст ошибки через `vim.notify(ERROR)` |

### `mdx.*` API в скриптах

```lua
mdx.notes()                       -- все заметки [{path, title}]
mdx.notes_today()                 -- заметки, изменённые сегодня
mdx.search_tags(include, exclude) -- поиск по тегам с wildcard'ами
mdx.query(sql, ...)               -- произвольный SELECT/WITH к БД
```

Все четыре — обёртки над LSP-методами `listNotes`, `searchByTags`, `query`. Если LSP не прицеплен — возвращают `{}` тихо. Можно использовать `vim.*` для дат, путей и прочего.

### Правила `to_string`

| Тип возврата | Что попадает в результат |
|---|---|
| `string` | как есть |
| `nil` | пустая строка → сообщение, выход |
| массив строк (`{"a", "b", ...}`) | `table.concat(\n)` |
| таблица записей или произвольная | `vim.inspect(...)` |
| прочее | `tostring(...)` |

### Пример

`~/.config/mdx/scripts/active-tasks.lua`:

```lua
-- description: Активные задачи списком
local rows = mdx.query(
  "SELECT path, COALESCE(title, '') AS title FROM notes "
  .. "WHERE json_extract(frontmatter, '$.type')   = 'task' "
  .. "  AND json_extract(frontmatter, '$.status') = 'active' "
  .. "ORDER BY title COLLATE NOCASE"
)
local lines = {}
for _, r in ipairs(rows) do
  table.insert(lines, "- [ ] " .. r.title)
end
return lines
```

`<leader>mr` → выбор `active-tasks` → Telescope с `- [ ] Foo`, `- [ ] Bar`.
`<leader>mR` → выбор того же скрипта → результат вставлен в текущий буфер построчно.

### Алгоритм

1. По опции `script_dir` найти каталог. Если нет — сообщение, выход.
2. Прочитать все `*.lua` файлы. У каждого — содержимое и опциональный `-- description: ...`.
3. Открыть picker скриптов (Telescope, file_previewer показывает .lua).
4. На выборе — `loadstring(source)`, `setfenv` с `mdx` и `vim`, `pcall`.
5. Привести результат к строке по правилам `to_string`.
6. Для `:MdxQuery` — разбить по `\n`, открыть picker строк (Enter закрывает picker).
7. Для `:MdxQueryInsert` — `nvim_put(lines, "c", true, true)` в позицию курсора.

### Размещение в коде

- `nvim/lua/mdx/script.lua` — парсинг, выполнение, оркестрация (resolve_dir, list, exec, to_string, query, query_insert).
- `nvim/lua/mdx/script_api.lua` — публичный `mdx.*` API, который подмешивается в окружение скрипта.
- `nvim/lua/mdx/picker.lua::pick_scripts`, `::pick_lines` — picker скриптов и picker отображения строк.
- `nvim/lua/mdx/query.lua` — клиент `mdx/query` LSP-метода (общий с `:MdxSql`).
- Тесты — `nvim/tests/script_spec.lua`.

### Заметки

- Скрипты пишутся пользователем в свой же каталог (как nvim-конфиг). Доверие соответствующее. Никакого автоисполнения — только по `:MdxQuery` / `:MdxQueryInsert`.
- Разница между `:MdxSql` и `:MdxQuery`: первый — голый SQL без программируемой обработки (результат всегда picker заметок, Enter открывает файл). Второй — Lua, можно строить любой текст. Если хочется навигации к заметке — `:MdxSql` удобнее; если нужен отчёт/сводка/format — `:MdxQuery`.

## `~/.config/mdx/ignore` — пути вне индекса

Файл-список путей, которые `mdx` исключает из индексации. Действует и на `mdx scan` (walker не спускается в исключённые каталоги), и на LSP-сервер (`textDocument/didOpen` и `textDocument/didSave` для файла под исключённым префиксом завершаются молча: ни upsert в БД, ни диагностики).

Чистка БД от уже проиндексированных записей под ignore-префиксами выполняется отдельной подкомандой `mdx gc` (см. `docs/strategy.md`): `scan` и `lsp` сами записи не удаляют, только отказываются добавлять/обновлять.

### Поведение

| Аспект | Поведение |
|---|---|
| Триггер | Запись `~/.config/mdx/ignore`. Никаких CLI-команд, никаких keymap'ов. |
| Расположение | `$XDG_CONFIG_HOME/mdx/ignore`, fallback `~/.config/mdx/ignore`. Переопределяется флагом `--ignore` и переменной `MDX_IGNORE`. Если файла нет — список путей пуст, поведение как до фичи. |
| Формат | Plain text, одна строка — один путь. Пустые строки и строки, начинающиеся с `#` (после возможного триминга), игнорируются. По краям строки триммится whitespace. |
| Поддерживаемые формы пути | Абсолютный (`/var/log`), `~/...` или одиночный `~` (разворачивается через `os.UserHomeDir`). Относительные пути не принимаются — строка пропускается, в stderr (для CLI) и в LSP-лог пишется запись уровня WARN с указанием файла и номера строки. |
| Нормализация | После разворачивания `~` путь чистится через `filepath.Clean`. Trailing slash убирается. |
| Семантика сравнения | Префиксное по нормализованному абсолютному пути. Файл `p` исключается, если для какого-либо префикса `q` из списка выполняется `p == q` либо `p` начинается с `q + string(filepath.Separator)`. Совпадение посередине пути не считается. |
| Эффект на `mdx scan` | Walker перед входом в каталог проверяет его путь против списка префиксов; матч → `fs.SkipDir`. Для не-директорий — та же проверка перед вызовом callback'а. Файлы под ignore-префиксом не передаются в pipeline; их строки в БД не создаются. |
| Эффект на LSP | В `onDidOpen` и `onDidSave` после `URIToPath(uri)` идёт проверка пути против ignore-префиксов. Матч → ранний return: `index.IndexBytes`/`IndexFile` не вызывается, `publishDiagnostics` не отправляется. В лог пишется DEBUG-запись `path ignored`. |
| Загрузка | Файл читается один раз — на старте `mdx scan` и на старте `mdx lsp`. Hot-reload (fsnotify, SIGHUP) — вне скоупа: ignore меняют редко, рестарт `mdx lsp` тривиален. |
| Уже проиндексированные файлы | `scan` и `lsp` записи не трогают — только перестают их обновлять. Чистка — задача `mdx gc`. |
| Ошибка чтения файла | `mdx scan` и `mdx lsp` стартуют как если бы файла не было, в stderr/лог пишется WARN с текстом ошибки. Стартовать с пустым списком безопаснее, чем падать. |

### Алгоритм

1. На старте `mdx scan` или `mdx lsp` резолвить путь к ignore-файлу: флаг `--ignore` → переменная `MDX_IGNORE` → `$XDG_CONFIG_HOME/mdx/ignore` → `~/.config/mdx/ignore`.
2. Загрузка через `config.LoadIgnore(path string) ([]string, error)`:
   - если файла нет — вернуть `nil, nil`;
   - построчно: триминг, пропуск пустых строк и строк, начинающихся с `#`;
   - разворачивание ведущих `~` / `~/`;
   - `filepath.Clean`;
   - отвержение оставшихся относительных путей с записью WARN;
   - возврат массива абсолютных префиксов.
3. В `cli.RunScan` передать список префиксов в `scan.Walk` дополнительным аргументом. `Walk` перед входом в каталог и перед callback'ом для файла проверяет путь через `config.IsIgnored(path string, prefixes []string) bool`.
4. В `lsp.RunLSP` загрузить список префиксов, сохранить в поле `Server.ignore`. Хендлеры `onDidOpen` и `onDidSave` после `URIToPath` вызывают `config.IsIgnored(path, s.ignore)` и при `true` возвращаются молча.

### Размещение в коде

- `lsp/internal/config/ignore.go` — новый пакет: `ResolveIgnorePath(flag string) (string, error)`, `LoadIgnore(path string) ([]string, error)`, `IsIgnored(path string, prefixes []string) bool`.
- `lsp/internal/config/ignore_test.go` — табличные тесты на `LoadIgnore` (комментарии, пустые строки, `~/...`, абсолют, относительный путь, отсутствующий файл) и на `IsIgnored` (точное совпадение, поддерево, не-поддерево с общим префиксом-как-строкой типа `/foo/barbaz` vs префикс `/foo/bar`).
- `lsp/internal/scan/walker.go` — расширение сигнатуры `Walk` параметром `ignorePrefixes []string` и проверкой перед входом в каталог и перед callback'ом для файла.
- `lsp/internal/scan/walker_test.go` — добавить кейс с ignore-префиксами.
- `lsp/internal/cli/scan.go` — `RunScan` принимает дополнительный аргумент `ignorePrefixes`, прокидывает в `Walk`.
- `lsp/internal/cli/lsp.go` — `RunLSP` загружает ignore через `config.LoadIgnore`, передаёт серверу.

- `lsp/cmd/mdx/main.go` — persistent-флаг `--ignore` (видимый и для `scan`, и для `lsp`), чтение `MDX_IGNORE` через `os.Getenv` в резолвере.

### Заметки

- Не gitignore-стиль: ни glob'ов (`**/state/`), ни отрицаний (`!path`), ни относительных паттернов. Линейный список абсолютных префиксов покрывает заявленный кейс (`~/.local/state`, отдельные подкаталоги) и не тащит парсер. Если со временем потребуются паттерны — отдельная задача.
- Существующий хардкод-список excludes по имени каталога (`.git`, `node_modules`, `.venv` и т.п.) сохраняется и работает независимо. Excludes — про базовые имена каталогов на любой глубине; ignore — про абсолютные пути. Пересечения нет.
- Ignore-механизм симметрично применяется к scan и к LSP: иначе после `mdx gc` исключённый каталог тут же возвращался бы в индекс через первый `didOpen` в этом подкаталоге.
- `mdx gc` чистит БД от записей, чьих файлов больше нет на диске или чьи пути попали под текущий список ignore. Никаких roots/excludes у gc нет: проверка идёт построчно по таблице `notes` (`os.Lstat` + `IsIgnored`).
