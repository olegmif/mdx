# M1 — LSP skeleton

## Цель

Получить работающий LSP-сервер `mdx lsp`, к которому neovim подключается через `lspconfig`. Сервер реагирует на жизненный цикл (`initialize`/`initialized`/`shutdown`/`exit`) и на два текстовых события — `textDocument/didOpen` и `textDocument/didSave`. На каждый из них он перепарсивает файл, обновляет записи в БД (через ту же логику, что и `mdx scan`, переиспользованную из общего пакета) и публикует diagnostics для битых ссылок.

После M1 у нас есть: (1) длинно живущий процесс, поддерживающий БД в актуальном состоянии для редактируемых в neovim файлов, (2) видимая обратная связь в редакторе про сломанные ссылки. На этот скелет в следующих milestone-ах навешиваются completion, code actions, workspace/symbol и т.п.

## Что входит и что не входит

### Входит

- Подкоманда `mdx lsp` — entry point LSP-сервера, общается по stdio.
- Минимальный JSON-RPC 2.0 транспорт с фреймингом `Content-Length` (или через выбранный фреймворк — см. ниже).
- Хендлеры lifecycle: `initialize`, `initialized`, `shutdown`, `exit`.
- Хендлеры `textDocument/didOpen`, `textDocument/didSave`.
- Извлечение из обоих хендлеров: парсинг файла, upsert в `notes`/`links`/`tags`, публикация diagnostics.
- Шаринг кода с CLI: общая функция «обработать один markdown-файл» вынесена в отдельный пакет, `mdx scan` и LSP вызывают её одинаково.
- Валидация ссылок: для каждой не-URL ссылки проверяется существование цели на диске. Если цели нет — diagnostic с `Severity = Warning`.
- Структурное логирование сервера в файл (`$XDG_STATE_HOME/mdx/lsp.log`) — stderr трогать нельзя, его читает клиент.
- Юнит-тесты на хендлеры (вызываем напрямую, без транспорта) и smoke-тест на полный round-trip через pipe.
- Раздел в `lsp/README.md` про запуск сервера и пример настройки `lspconfig`.

### Не входит (отложено)

- `textDocument/didChange` — incremental или full sync. На M1 БД обновляется только на open/save. Live-обновление между нажатиями клавиш не нужно для MVP, добавится позже без поломки протокола (в `initialize` declare-им только то, что реализуем).
- `textDocument/completion` (M3 — picker и вставка ссылок).
- `textDocument/codeAction` (M7 — починка битых ссылок).
- `workspace/symbol`, `workspace/didChangeWatchedFiles` (M4, M6).
- Конкурентная обработка нескольких didSave одновременно для разных файлов: на M1 — последовательная очередь. SQLite WAL это переживает, упростит отладку.
- Watcher для файлов, не открытых в neovim (M6 — `mdx gc`).
- Resolve якорей внутри файлов (`#section`) — пока считаем валидной любую ссылку, у которой существует целевой файл.
- Прогрев БД на `initialize` (полный `scan` инициируется руками через CLI).

## Технические решения, которые надо зафиксировать перед стартом

| Вопрос | Предлагаемое решение |
|---|---|
| LSP-фреймворк | **`github.com/tliron/glsp`** + `glsp/server` — даёт типы протокола (`protocol_3_16`), готовый stdio-транспорт и роутинг по методам. Альтернатива: голый `jsonrpc2` + ручной диспатч; полезно как упражнение, но вытаскивает в M1 много шаблонного кода (фрейминг, типы, заглушки) и удлиняет milestone. **Берём glsp**, при упоре во что-то — мигрируем (типы LSP стандартные, переезд механический). |
| Транспорт | stdio (`os.Stdin`/`os.Stdout`). Это то, чего ждёт `lspconfig` по умолчанию. TCP/socket — не нужно. |
| Шаринг кода между CLI и LSP | Вынести «обработать один файл и записать в БД» из `internal/cli/scan.go` в новый пакет `internal/index` с функцией `IndexFile(ctx, db, path) (Result, error)`. CLI зовёт её в цикле, LSP — на каждый didOpen/didSave. |
| Парсинг текста, переданного LSP | На didOpen приходит `text` уже в payload — парсим прямо его, без чтения файла с диска (так увидим несохранённые правки). На didSave файл уже сохранён — читаем с диска (плюс пересчитываем hash/mtime/size). |
| Стратегия diagnostics | Публикуем после каждого индексирующего события на тот URI, который индексировали. Полное замещение, не инкремент. |
| Severity для битой ссылки | `DiagnosticSeverity.Warning`. Это не синтаксическая ошибка markdown'а — файл валиден, просто ссылка не разрешается. |
| Источник правды для проверки ссылки | `os.Lstat(targetPath)`. БД не используем — она может отставать. Дешевле и надёжнее. |
| Концерн «БД из CLI и БД из LSP — одна и та же» | Да. WAL разрешает одновременных читателей и одного писателя; LSP и `mdx scan` могут работать параллельно. Проверим smoke-тестом. |
| URI ↔ путь | Только `file://` URI. Конвертация: `net/url` + `filepath`. Не-`file` URI отклоняем с логом и без crash'а. |
| Логирование | `log/slog` с JSON-handler-ом в файл. Уровень — `INFO` по умолчанию, `DEBUG` через `MDX_LOG=debug`. |
| Конкурентность хендлеров | Один глобальный `sync.Mutex` вокруг секции «прочитать/распарсить/записать в БД» — на M1 этого хватит и упрощает рассуждение. Снимем, когда профиль покажет, что это узкое место. |
| Версия LSP | `protocol_3_16` (то же, что использует `lspconfig` по умолчанию). |

## Изменения в существующем коде

- `internal/cli/scan.go::scanOne` — переименовывается, выезжает в `internal/index/index.go::IndexFile`. Сигнатура: `func IndexFile(ctx context.Context, conn *sql.DB, path string) (Result, error)`. Дополнительный конструктор `IndexBytes(ctx, conn, path, content []byte) (Result, error)` — для LSP, где содержимое уже на руках.
- `Result` несёт то, что нужно diagnostics: список ссылок с резолвленными путями и позициями. Нужно, чтобы не парсить файл дважды.
- `internal/cli/scan.go::Run` — становится тонкой обёрткой: `Walk` → `index.IndexFile`.
- `cmd/mdx/main.go` — регистрируется новая команда `lsp` рядом с `scan`.

Других изменений в существующих пакетах не требуется. БД-схема та же, что в M0.

## Контракт LSP-сервера

### Объявляемые capabilities (в ответе на `initialize`)

```jsonc
{
  "capabilities": {
    "textDocumentSync": {
      "openClose": true,
      "change": 0,           // None — didChange не обрабатываем
      "save": { "includeText": false }
    }
  },
  "serverInfo": { "name": "mdx", "version": "<git describe>" }
}
```

`includeText: false` — на didSave мы перечитываем с диска сами, лишний трафик не нужен.

### Поведение хендлеров

- **`initialize`** — открыть БД (по тем же правилам, что и CLI: `MDX_DB` → `--db` → XDG), применить миграции, поднять логгер. Если БД недоступна — вернуть LSP-ошибку, neovim покажет её и закроет соединение.
- **`initialized`** — no-op.
- **`textDocument/didOpen`** — `IndexBytes(uri→path, params.text)`, потом `publishDiagnostics`.
- **`textDocument/didSave`** — `IndexFile(uri→path)` (читаем с диска), потом `publishDiagnostics`.
- **`shutdown`** — закрыть БД, ничего не отвечать кроме `null`.
- **`exit`** — `os.Exit(0)` (или `1`, если до этого не было `shutdown`).

### Diagnostics

Для каждой ссылки, у которой `RawTarget` не URL и `Lstat(TargetPath)` дал `ErrNotExist`:

```
range:    позиция [text](target) в файле — берём `Line`, `Col` начала из парсера
severity: Warning
source:   "mdx"
message:  "broken link: <RawTarget>"
```

Если ссылок нет или все валидны — публикуем пустой массив (это очищает прежние diagnostics для URI).

## Шаги выполнения

Шаги упорядочены так, чтобы каждый оставлял систему собранной и тестируемой.

### Шаг 1. Выделить пакет `internal/index`

- Создать `internal/index/index.go`.
- Перенести `scanOne` и `resolveTarget` сюда. Превратить в:
  - `IndexFile(ctx, conn, path) (Result, error)` — читает файл с диска, считает hash/stat, делегирует в `IndexBytes`.
  - `IndexBytes(ctx, conn, path, content) (Result, error)` — основная логика. Hash/stat для пути берёт сам (mtime/size — `os.Lstat`, hash — `sha256` от `content`).
- `Result` содержит: `Links []ResolvedLink{ RawTarget, TargetPath, Line, Col }`. Поля `Tags`/`Title` — по необходимости, на M1 для diagnostics не нужны.
- Поправить `internal/cli/scan.go::Run`: вызывает `index.IndexFile` вместо локального `scanOne`.
- Прогнать `go test ./...` — все существующие тесты должны зеленеть без правок (логика та же, только переезжала). `e2e_test` проверит, что результат скана не изменился.

**Готовность:** `make test` зелёный, поведение `mdx scan` неизменно.

### Шаг 2. Подключить glsp как зависимость

- `go get github.com/tliron/glsp@latest` и `github.com/tliron/glsp/server`.
- В `internal/lsp/server.go` — пустой `New(conn *sql.DB) *server.Server`, который пока возвращает `glsp.Handler` без зарегистрированных методов.

**Готовность:** `make build` собирает, `go vet ./...` чист.

### Шаг 3. Подкоманда `mdx lsp`

- `internal/cli/lsp.go::Run(ctx, conn) error` — берёт `*server.Server`, запускает `srv.RunStdio()`. Блокируется до закрытия соединения.
- В `cmd/mdx/main.go` — добавить cobra-команду `lsp`. У неё те же глобальные флаги (`--db`), плюс `--log` (путь к лог-файлу, дефолт — XDG state dir).
- В этой же команде — резолвинг и инициализация БД (как в `scan`), затем вызов `cli.LSPRun`.

**Готовность:** `./bin/mdx lsp` запускается, ничего не делает, ждёт вход в stdin. Закрытие stdin → graceful exit.

### Шаг 4. Логирование

- `internal/lsp/log.go`: открыть файл по пути из флага/env, обернуть в `slog.NewJSONHandler`, сохранить как глобальный логгер (`slog.SetDefault`).
- Логгировать в стартапе: версию, путь к БД, путь к лог-файлу, capabilities.
- Все хендлеры пишут через `slog.InfoContext` с request-id (`ctx`-key — добавим в Шаге 5).

**Готовность:** `mdx lsp` создаёт лог-файл, при старте пишет одну строку `server starting`.

### Шаг 5. Хендлеры lifecycle

- `internal/lsp/handlers.go::onInitialize(ctx, params) (*InitializeResult, error)` — заполняет capabilities (см. контракт выше), пишет в лог, возвращает.
- `onInitialized` — no-op + лог.
- `onShutdown` — закрывает БД, ставит флаг `shutdownRequested`.
- `onExit` — если флаг не выставлен → `os.Exit(1)`, иначе `os.Exit(0)`.
- Зарегистрировать их через `protocol_3_16.Handler{...}.CreateServer()` (точное имя смотрим в glsp).
- Юнит-тест: вызываем `onInitialize` напрямую, проверяем, что `TextDocumentSync.OpenClose == true` и `Save.IncludeText == false`.

**Готовность:** `nvim` с настроенным `lspconfig` цепляется к серверу — в `:LspInfo` видно, что mdx attached.

### Шаг 6. URI helper

- `internal/lsp/uri.go::URIToPath(uri string) (string, error)` — `url.Parse`, проверка `scheme == "file"`, `filepath.FromSlash` для пути.
- `PathToURI(path string) string` — обратное.
- Тесты: `file:///home/x/y.md` ↔ `/home/x/y.md`, отказ на `http://...`.

**Готовность:** unit-тесты зелёные.

### Шаг 7. didOpen

- `onDidOpen(ctx, params)`:
  1. URI → path.
  2. `index.IndexBytes(ctx, conn, path, []byte(params.TextDocument.Text))` под глобальным мьютексом.
  3. Построить диагностики из `Result.Links` (см. Шаг 9).
  4. Отправить `textDocument/publishDiagnostics`.
- Юнит-тест: подсунуть моки/in-memory SQLite, вызвать с фикстурным content, проверить, что в БД появился note и в notification — ожидаемые диагностики.

**Готовность:** при открытии `.md` файла в neovim в `:LspLog` виден didOpen, в `:diagnostics` — битые ссылки (если есть).

### Шаг 8. didSave

- `onDidSave(ctx, params)` — то же, но через `index.IndexFile` (читаем с диска).
- Тест аналогичный.

**Готовность:** правка-сохранение `.md` в neovim → диагностики обновляются.

### Шаг 9. Сборщик диагностик

- `internal/lsp/diagnostics.go::Build(result index.Result) []protocol.Diagnostic`.
- Логика: для каждой ссылки — пропустить URL-схемы (http/https/mailto/...) и якоря (`#...`), на остальных — `Lstat`. Если `IsNotExist(err)` → `Diagnostic{...}`.
- Range на M1 — однопозиционный (`start == end == {Line, Col}`); полную ширину «токена ссылки» считаем при необходимости позже (нужны парсеру дополнительные поля).
- Тесты: фикстура с битыми и валидными ссылками, проверяем количество и месседжи.

**Готовность:** unit-тесты зелёные.

### Шаг 10. Глобальный мьютекс и graceful shutdown

- В `lsp.Server` хранить `sync.Mutex`. Все хендлеры, изменяющие БД или публикующие диагностики для одного URI, держат его на время операции.
- На `shutdown` — закрытие БД с ожиданием активных транзакций (Go это делает за нас, если мы корректно завершаем `Begin/Commit`).
- На сигналы `SIGINT/SIGTERM` — то же, что shutdown+exit (через `signal.NotifyContext`).

**Готовность:** `kill -INT $(pgrep mdx)` → процесс выходит без панических трейсов в логе.

### Шаг 11. Smoke-тест round-trip

- `internal/lsp/handlers_test.go` или отдельный `e2e_test.go`:
  - Запускаем сервер не через `RunStdio`, а через ручной transport: пара `io.Pipe`, в которые мы пишем готовые JSON-RPC сообщения (`initialize` → `didOpen` → `shutdown` → `exit`).
  - Парсим ответы, ассертим: `InitializeResult` валиден, на didOpen прилетел `publishDiagnostics` с нужными элементами.
- Используем фикстуру `testdata/fixtures` из M0.

**Готовность:** `make test` зелёный.

### Шаг 12. Документация и пример конфига

- В `lsp/README.md` добавить раздел «LSP»: как запускать (`mdx lsp`), путь к логу, минимальный snippet `lspconfig`:

  ```lua
  require('lspconfig.configs').mdx = {
    default_config = {
      cmd = { 'mdx', 'lsp' },
      filetypes = { 'markdown' },
      root_dir = function() return vim.loop.cwd() end,
    },
  }
  require('lspconfig').mdx.setup{}
  ```

- Указать, что сервер обновляет БД только на open/save и что для batch-индексации существует `mdx scan`.

**Готовность:** README покрывает запуск из nvim.

## Критерии готовности M1

1. `make build` собирает бинарь.
2. `make test` зелёный, включая smoke-тест round-trip.
3. `mdx lsp` запускается; в neovim с настроенным `lspconfig` команда `:LspInfo` показывает `mdx` среди attached серверов.
4. После `:edit some.md` (с битой ссылкой) в `:diagnostics` появляется warning от source `mdx` с месседжем `broken link: <target>`.
5. После `:write` БД содержит обновлённую запись для этого файла (проверяется sqlite-клиентом или повторным `mdx scan` без diff'а).
6. CLI `mdx scan` всё ещё работает и даёт тот же результат, что в M0 (поведение не сломали при выделении `internal/index`).
7. Лог-файл создаётся и содержит структурированные записи; stderr сервера пуст.
8. Закрытие neovim → процесс `mdx lsp` завершается (по `shutdown`/`exit` от клиента).

После этих пунктов M1 закрыт, переходим к M2 (neovim-плагин: навигация).

## Что нужно подтвердить, прежде чем стартовать

1. **LSP-фреймворк** — `glsp` (предлагаемый по умолчанию) или ручная реализация поверх `jsonrpc2`. Влияет на Шаги 2, 5, 7, 8, 11.
2. **Lock-стратегия** — глобальный мьютекс на M1 vs. сразу per-URI lock. По умолчанию — глобальный (см. таблицу), снимем позже.
3. **Severity для битой ссылки** — `Warning` (предложено) или `Error`. Зависит от того, насколько громко ты хочешь видеть это в `:diagnostics`.
4. **Куда писать лог по умолчанию** — `$XDG_STATE_HOME/mdx/lsp.log` (предложено) или рядом с БД (`$XDG_DATA_HOME/mdx/lsp.log`). XDG-spec: `state` — корректнее, это «runtime state».
