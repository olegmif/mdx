# M1 — LSP skeleton

## Цель

После M1 при открытии `.md`-файла в neovim наш бинарь видит этот факт и сразу показывает в редакторе предупреждения на ссылки, которые не разрешаются на диске. На каждое сохранение файла соответствующая запись в БД переиндексируется. Под капотом — длинно живущий процесс `mdx lsp`, которого neovim запускает как дочерний и общается с ним по LSP через stdin/stdout.

Это даёт нам две вещи:

1. Скелет протокола, на который в M2+ навешиваются follow-link, completion, code actions, поиск по тегам.
2. Видимую обратную связь в редакторе — основное свойство, ради которого LSP-сервер вообще нужен.

## Что входит и что не входит

### Входит

- Подкоманда `mdx lsp`, общающаяся с клиентом через stdin/stdout.
- Реакция на жизненный цикл протокола: `initialize`, `initialized`, `shutdown`, `exit`.
- Реакция на `textDocument/didOpen` и `textDocument/didSave` с переиндексацией файла и публикацией диагностик через `textDocument/publishDiagnostics`.
- Общий пакет `internal/index` для логики «обработать один файл», который вызывают и CLI, и LSP. Это единственное структурное изменение, которое M1 вносит в существующий код M0.
- Структурное логирование сервера в файл (`$XDG_STATE_HOME/mdx/lsp.log`).
- Юнит-тесты на каждый обработчик и smoke-тест полного round-trip через `io.Pipe`.
- Раздел в `lsp/README.md` про запуск сервера и пример конфигурации `lspconfig`.

### Не входит (отложено)

- `textDocument/didChange` — обновление БД на каждое нажатие клавиши. На M1 БД обновляется только на open и save; это достаточно для MVP и упрощает рассуждение о согласованности. Добавим позже без поломки существующего поведения: capabilities в `initialize` декларируют только то, что мы реально реализуем.
- `textDocument/completion`, `textDocument/codeAction`, `workspace/symbol` — уезжают в M3, M7, M4 соответственно.
- Watcher для файлов, которые редактируются мимо neovim. Эту дыру закрывает `mdx gc` в M6.
- Резолв якорей вида `#section` внутри `.md`. На M1 ссылка считается валидной, если существует целевой файл; раздел внутри файла мы не проверяем.
- Прогрев БД на старте сервера. Полное сканирование запускается отдельной командой `mdx scan`.
- Параллельная обработка нескольких сообщений: на M1 одна общая блокировка вокруг индексации и публикации диагностик. Снимем по необходимости.

## Технические решения, которые надо зафиксировать перед стартом

| Вопрос | Решение |
|---|---|
| Библиотека протокола | `github.com/tliron/glsp` берёт на себя фрейминг `Content-Length`, разбор JSON-RPC и маршрутизацию по имени метода в зарегистрированный обработчик. Также поставляет типы всех сообщений (`InitializeParams`, `Diagnostic` и т.д.), что избавляет от написания собственных DTO. Альтернатива — сделать всё вручную поверх `jsonrpc2`; это около пятисот строк инфраструктурного кода, не относящегося к содержанию milestone-а. |
| Транспорт | stdio. Это то, чего ожидает `lspconfig` по умолчанию; других режимов M1 не предусматривает. |
| Версия LSP | `protocol_3_16` — стандартный диалект для современного `lspconfig`. В glsp выбирается импортом подпакета. |
| Где живёт логика «обработать один файл» | В новом пакете `internal/index`. Туда переезжают `cli.scanOne` и `cli.resolveTarget`. CLI и LSP импортируют один и тот же пакет. |
| Источник содержимого на didOpen | `params.TextDocument.Text`, переданный клиентом. Может отличаться от того, что лежит на диске (несохранённый буфер); правильнее доверять клиенту. |
| Источник содержимого на didSave | Чтение с диска. На момент save буфер записан, файл — авторитетный источник. |
| Проверка ссылки на «битость» | `os.Lstat(targetPath)`. БД для этого не используется: она может отставать (например, целевой файл только что создан в другом редакторе). |
| Severity для битой ссылки | `Warning`. Файл синтаксически корректен; недоступная ссылка — не ошибка markdown'а. |
| URI ↔ путь | Только `file://` URI; перевод через `net/url` и `filepath.FromSlash`. URI с другими схемами игнорируются с записью в лог. |
| Логирование | `log/slog` с JSON-handler-ом, пишущим в файл `$XDG_STATE_HOME/mdx/lsp.log`. Уровень `INFO` по умолчанию, `DEBUG` при `MDX_LOG=debug`. |
| Конкурентность | Один глобальный `sync.Mutex` вокруг индексирующей секции и отправки диагностик. На M1 этого достаточно. |

## Шаги выполнения

Шаги упорядочены так, чтобы после каждого можно было запустить осмысленную проверку — либо тест, либо ручной сценарий.

### Шаг 1. Выделить пакет `internal/index`

Вынести логику обработки markdown-файла из `internal/cli/scan.go` в отдельный пакет, доступный для вызова из CLI и из LSP-обработчиков.

#### Состав пакета `internal/index`

`index.go` объявляет следующие типы:

```go
type ResolvedLink struct {
    RawTarget  string  // целевой путь, как записан в исходнике
    TargetPath string  // абсолютный путь после резолва
    Line       int     // строка ссылки (1-based)
    Col        int     // колонка ссылки (1-based, в байтах)
}

type Result struct {
    Path  string         // абсолютный путь индексированного файла
    Links []ResolvedLink // исходящие ссылки на не-URL цели
}
```

И следующие функции:

```go
func IndexBytes(ctx context.Context, conn *sql.DB, path string, content []byte) (Result, error)
```

Принимает содержимое файла в памяти. Парсит frontmatter и тело, извлекает ссылки и теги, резолвит относительные пути в абсолютные относительно `filepath.Dir(path)`. Вычисляет SHA-256 от `content`. Берёт `mtime`, `size`, `inode` через `scan.Stat(path)`. Записывает одной транзакцией в таблицы `notes`, `links`, `tags`. Возвращает `Result` со списком разрешённых ссылок.

```go
func IndexFile(ctx context.Context, conn *sql.DB, path string) (Result, error)
```

Читает файл с диска через `os.ReadFile(path)` и делегирует в `IndexBytes`.

`resolve.go` — функция `resolveTarget(sourceDir, raw string) string`, перенесённая из `cli` без изменений.

`index_test.go` — табличный тест `TestResolveTarget`, перенесённый из `cli`.

#### Изменения в `internal/cli`

- В `scan.go` тело цикла `Run` сводится к вызову `index.IndexFile`. Удаляются локальные функции `scanOne` и `resolveTarget`, а также импорты, ставшие ненужными: `encoding/json`, `path/filepath`, `strings`, пакеты `db` и `parse`.
- Файл `scan_test.go` удаляется целиком — его единственный тест перенесён в `index`.

**Готовность:** `go test ./...` зелёный, e2e-тест `internal/cli` проходит через `index.IndexFile`, поведение `mdx scan` снаружи не изменилось.

### Шаг 3. Подкоманда `mdx lsp` и логирование

Зарегистрировать cobra-команду `lsp`, поднять логирование в файл и запустить LSP-сервер на stdio.

#### Состав пакета `internal/lsp`

`log.go` объявляет функции:

```go
func Init(path string) error
func ResolveLogPath(flag string) (string, error)
```

`Init` создаёт каталог при необходимости, открывает файл на запись в режиме `O_CREATE|O_WRONLY|O_APPEND`, оборачивает его в `slog.NewJSONHandler`, устанавливает результирующий логгер как `slog.SetDefault`. Уровень — `INFO` по умолчанию, `DEBUG` если переменная окружения `MDX_LOG` равна `debug`.

`ResolveLogPath` возвращает путь к лог-файлу по следующему приоритету: значение `flag`, если непустое; иначе `$XDG_STATE_HOME/mdx/lsp.log`, если переменная установлена; иначе `$HOME/.local/state/mdx/lsp.log`.

#### Состав пакета `internal/cli`

Существующая функция `Run` в `scan.go` переименовывается в `RunScan` во избежание коллизии с новой функцией для LSP. Соответствующие вызовы обновляются в `cmd/mdx/main.go` и `internal/cli/e2e_test.go`.

`lsp.go` объявляет функцию:

```go
func RunLSP(ctx context.Context, conn *sql.DB, dbPath, logPath string) error
```

Вызывает `lsp.Init(logPath)`, эмитит первую запись в лог (`server starting` с полями `version`, `db`, `log`; версия читается через `runtime/debug.ReadBuildInfo()`, fallback `"dev"`), конструирует сервер через `lsp.New(conn)`, запускает `srv.RunStdio()`, блокируется до закрытия соединения.

#### Изменения в `cmd/mdx/main.go`

- Флаг `--db` переносится из локального флага `scanCmd` в `rootCmd.PersistentFlags()`, чтобы команда `lsp` его тоже видела.
- Регистрируется cobra-команда `lsp` с локальным флагом `--log` (путь к файлу логов).
- Добавляется обработчик `runLSP`: резолвит `dbPath` через `db.ResolvePath`, открывает БД, мигрирует, резолвит `logPath` через `lsp.ResolveLogPath`, оборачивает контекст в `signal.NotifyContext(ctx, os.Interrupt)`, вызывает `cli.RunLSP`.

**Готовность:** `./bin/mdx lsp` запускается и блокируется на чтении stdin. В лог-файле появляется первая структурированная запись со стартовой информацией — версия бинаря, путь к БД, путь к самому логу. Закрытие stdin приводит к корректному завершению процесса без trace.

### Шаг 4. Обработчики жизненного цикла

Реализовать обработку `initialize`, `initialized`, `shutdown`, `exit` и навесить их на структуру `protocol_3_16.Handler`, передаваемую в `glsp.NewServer`.

#### Состав `internal/lsp/handlers.go`

```go
func onInitialize(ctx *glsp.Context, params *protocol.InitializeParams) (any, error)
func onInitialized(ctx *glsp.Context, params *protocol.InitializedParams) error
func onShutdown(ctx *glsp.Context) error
func onExit(ctx *glsp.Context) error
```

`onInitialize` возвращает `InitializeResult` с capabilities: `TextDocumentSync.OpenClose = true`, `TextDocumentSync.Change = TextDocumentSyncKindNone`, `TextDocumentSync.Save.IncludeText = false`. Поле `ServerInfo` заполняется именем `mdx` и версией.

`onInitialized` записывает в лог факт хендшейка.

`onShutdown` закрывает соединение с БД и выставляет внутренний флаг `shutdownReceived`.

`onExit` проверяет флаг: если выставлен — вызывает `os.Exit(0)`, иначе `os.Exit(1)`.

#### Тесты

`handlers_test.go` вызывает `onInitialize` напрямую с фиктивным `params` и сверяет полученные capabilities с ожидаемыми.

**Готовность:** в neovim с настроенным `lspconfig` команда `:LspInfo` показывает `mdx` среди attached серверов; в лог-файле виден успешно завершённый хендшейк.

### Шаг 5. Хелпер URI ↔ путь

Реализовать перевод между `file://`-URI и путями файловой системы.

#### Состав `internal/lsp/uri.go`

```go
func URIToPath(uri string) (string, error)
func PathToURI(path string) string
```

`URIToPath` парсит URI через `url.Parse`, проверяет, что `Scheme == "file"`, возвращает `filepath.FromSlash(u.Path)`. На любую другую схему возвращает ошибку.

`PathToURI` строит URI обратно: `(&url.URL{Scheme: "file", Path: filepath.ToSlash(path)}).String()`.

#### Тесты

`uri_test.go` проверяет round-trip на нескольких парах путь/URI и факт ошибки на `http://...`.

**Готовность:** unit-тесты зелёные.

### Шаг 6. Сборка диагностик и глобальная блокировка

Реализовать построение списка `protocol.Diagnostic` из списка разрешённых ссылок и подготовить серверную инфраструктуру для индексирующих обработчиков (общий мьютекс).

#### Состав `internal/lsp/diagnostics.go`

```go
func Build(links []index.ResolvedLink) []protocol.Diagnostic
```

Для каждой ссылки выполняет `os.Lstat(link.TargetPath)`. Если `errors.Is(err, fs.ErrNotExist)`, добавляет в результат диагностику со следующими полями:

- `Range`: однопозиционный — `start == end == { Line: link.Line - 1, Character: link.Col - 1 }`. LSP считает позиции с нуля, парсер — с единицы.
- `Severity`: `DiagnosticSeverityWarning`.
- `Source`: `"mdx"`.
- `Message`: `fmt.Sprintf("broken link: %s", link.RawTarget)`.

Полная ширина «токена ссылки» (от `[` до `)`) на M1 не вычисляется — парсер отдаёт только начальную позицию.

#### Изменения в `internal/lsp/server.go`

В структуру `Server` добавляется поле `mu sync.Mutex`. Индексирующие обработчики из шагов 7 и 8 берут его в начале вызова и отпускают в конце.

#### Тесты

`diagnostics_test.go` подаёт списки `ResolvedLink` с битыми и валидными целями в `t.TempDir()` и сверяет получаемый список диагностик с ожидаемым.

**Готовность:** unit-тесты зелёные.

### Шаг 7. Обработчик didOpen

Реализовать переиндексацию открываемого файла и публикацию диагностик по содержимому, переданному клиентом в payload.

#### Состав `internal/lsp/handlers.go`

```go
func (s *Server) onDidOpen(ctx *glsp.Context, params *protocol.DidOpenTextDocumentParams) error
```

Извлекает URI и текст из `params.TextDocument`. Через `URIToPath` получает путь файловой системы. Под `s.mu` вызывает `index.IndexBytes(context.Background(), s.conn, path, []byte(text))`. Передаёт `Result.Links` в `Build`, получает `[]protocol.Diagnostic`. Отправляет уведомление `textDocument/publishDiagnostics` через `ctx.Notify`.

Ошибки на любом из шагов логируются и публикуются как пустой набор диагностик; сервер не падает.

В `lsp.New` обработчик навешивается на поле `protocol.Handler.TextDocumentDidOpen`.

#### Тесты

`handlers_test.go` использует БД во временном каталоге. Подаётся фиктивный `DidOpenTextDocumentParams` со ссылкой на несуществующий файл; проверяется, что (а) в таблице `notes` появилась строка для соответствующего пути, (б) была отправлена нотификация ровно с одной диагностикой ожидаемого вида.

**Готовность:** при выполнении `:edit some.md` в neovim в логе сервера появляется запись о didOpen, в `:diagnostic` — записи на каждой битой ссылке открытого файла.

### Шаг 8. Обработчик didSave

Реализовать переиндексацию сохранённого файла по содержимому с диска.

#### Состав `internal/lsp/handlers.go`

```go
func (s *Server) onDidSave(ctx *glsp.Context, params *protocol.DidSaveTextDocumentParams) error
```

Повторяет схему `onDidOpen`, но вместо `index.IndexBytes` вызывает `index.IndexFile(context.Background(), s.conn, path)`. Обработчик навешивается на поле `protocol.Handler.TextDocumentDidSave`.

#### Тесты

`handlers_test.go` эмулирует save файла из фикстур; проверяется обновлённая запись в `notes` и отправленная нотификация.

**Готовность:** в neovim последовательность правка → `:write` приводит к обновлённому набору диагностик, а строка в `notes` отражает свежее содержимое файла (mtime, size, content_hash).

### Шаг 9. Graceful shutdown

Корректно обрабатывать сигналы завершения процесса так, чтобы БД закрывалась и WAL-файлы свёртывались.

#### Изменения в `cmd/mdx/main.go`

В обработчике команды `lsp` корневой контекст оборачивается:

```go
ctx, stop := signal.NotifyContext(context.Background(), os.Interrupt, syscall.SIGTERM)
defer stop()
```

Отмена контекста приводит к завершению `RunStdio` и срабатыванию deferred `conn.Close()`. Перед выходом в лог пишется запись `server stopped`.

**Готовность:** `kill -INT $(pgrep mdx)` приводит к завершению процесса; в логе появляется строка `server stopped`; БД-файл не остаётся в WAL-режиме (`*-wal` и `*-shm` файлы свёрнуты).

### Шаг 10. Smoke-тест round-trip

Реализовать end-to-end тест, прогоняющий полный цикл сообщений через настоящий транспорт сервера.

#### Состав `internal/lsp/e2e_test.go`

Создаются две пары `io.Pipe` для имитации stdin/stdout сервера. Сервер запускается в отдельной горутине с этими пайпами в роли стандартных потоков. С тестового конца последовательно записываются корректно фреймированные сообщения:

```
initialize → initialized → didOpen → shutdown → exit
```

На didOpen подаётся фикстура из `testdata/fixtures`, содержащая битую ссылку. С тестового конца читаются: `InitializeResult` (response с тем же `id`) и `publishDiagnostics` (notification с непустым списком). Распарсенные структуры сверяются с ожидаемыми.

Тест не зависит от установленного neovim и не запускает дополнительных процессов.

**Готовность:** `make test` зелёный, тест воспроизводимый.

**Готовность:** README отвечает на вопрос «как запустить mdx из neovim» в рамках одного экрана.

## Критерии готовности M1

1. `make build` собирает бинарь без ошибок.
2. `make test` зелёный, включая smoke-тест round-trip.
3. `mdx lsp` запускается; в neovim с настроенным `lspconfig` команда `:LspInfo` показывает `mdx` среди attached серверов.
4. После `:edit some.md` (где есть битая ссылка) в `:diagnostic` появляется warning с источником `mdx` и сообщением `broken link: <target>`.
5. После `:write` запись в БД для этого файла обновлена; повторный `mdx scan` не находит, что переиндексировать.
6. CLI `mdx scan` ведёт себя так же, как в M0 — выделение пакета `internal/index` не повлияло на наблюдаемое поведение.
7. Лог-файл создаётся и содержит структурированные записи; в stderr сервер ничего не пишет.
8. Закрытие neovim приводит к завершению процесса `mdx lsp` через корректную последовательность `shutdown` + `exit`.

После выполнения этих пунктов M1 закрыт, переходим к M2 (neovim-плагин: навигация по ссылкам).

## Что нужно подтвердить, прежде чем стартовать

1. **Библиотека протокола.** По умолчанию — glsp. Альтернатива — реализация поверх `jsonrpc2`. Влияет на Шаги 2, 4, 6, 7, 10.
2. **Severity для битой ссылки.** Предложено `Warning`. Альтернатива — `Error`, если хочется, чтобы такие ссылки заметнее всплывали в `:diagnostic`.
3. **Путь к лог-файлу.** Предложено `$XDG_STATE_HOME/mdx/lsp.log` (по XDG-spec логи относятся к state). Альтернатива — рядом с БД в `$XDG_DATA_HOME/mdx/lsp.log`.
