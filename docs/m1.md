# M1 — LSP skeleton

## Цель

Сделать так, чтобы при открытии `.md`-файла в neovim наш бинарь видел этот факт, обновлял БД и подсвечивал битые ссылки прямо в редакторе. Внутри — длинно живущий процесс `mdx lsp`, который neovim запускает сам и говорит с ним по протоколу LSP.

После M1 у нас:
1. Фундамент протокола — на него в M2+ навесим follow-link, completion, code actions.
2. Видимая обратная связь в редакторе: открыл файл с битой ссылкой — увидел warning.
3. БД остаётся актуальной для редактируемых файлов: каждый save → переиндексация.

## Что такое LSP — простыми словами

Если раньше с языковыми серверами не работал — эту секцию пропускать нельзя. Дальше всё предполагает, что ты понимаешь эти штуки.

### Кто кого запускает

В neovim в конфиге пишут: «для markdown-файлов запускай команду `mdx lsp`». Как только пользователь открывает первый `.md`, neovim сам запускает `mdx lsp` как дочерний процесс. Между ними образуется пара пайпов: `stdin` (neovim → сервер) и `stdout` (сервер → neovim). Через них летят сообщения. Сервер живёт, пока neovim его не убьёт. Открыл десять `.md` — всё равно один процесс `mdx lsp`.

`stderr` для бизнес-данных не используем — туда не пишем вообще, или пишем только аварии. Свои логи льём в файл, потому что некоторые LSP-клиенты считают любой вывод в stderr признаком сбоя.

### Как они общаются: JSON-RPC поверх stdio

Каждое сообщение — JSON, обёрнутый в HTTP-подобный конверт:

```
Content-Length: 123\r\n
\r\n
{"jsonrpc":"2.0","method":"textDocument/didOpen","params":{...}}
```

Эта обёртка называется **фрейминг**. Читать/писать её будет за нас библиотека (см. ниже). Внутри — обычный JSON-RPC 2.0, в нём бывают два типа сообщений:

- **Request** — у него есть поле `id`, отправитель ждёт ответа с тем же `id`. Пример: `initialize`.
- **Notification** — без `id`, ответа не ждёт, как fire-and-forget. Пример: `textDocument/didOpen`.

### Жизненный цикл

Каждая LSP-сессия начинается и заканчивается одинаково:

```
neovim                            mdx lsp
   │                                  │
   │── initialize (request) ─────────>│   "вот корень рабочего каталога,
   │                                  │    что ты умеешь?"
   │<── InitializeResult ─────────────│   "вот мой список умений (capabilities)"
   │── initialized (notif) ──────────>│   "ок, начинаем"
   │                                  │
   │   ... обычная работа ...         │
   │                                  │
   │── shutdown (request) ───────────>│   "готовься к выходу"
   │<── null ─────────────────────────│
   │── exit (notif) ─────────────────>│
   │                                  ✗   процесс делает exit(0)
```

«Обычная работа» в нашем случае на M1 — это:

```
   │── didOpen (notif)  ─────────────>│   "пользователь открыл x.md, вот его текст"
   │<── publishDiagnostics (notif) ───│   "в этом файле такие-то проблемы"
   │                                  │
   │── didSave (notif)  ─────────────>│   "пользователь сохранил x.md"
   │<── publishDiagnostics (notif) ───│   "обновлённый список проблем"
```

`didOpen`/`didSave` — это нотификации (без ответа). Сервер на них реагирует **сам по себе**, отправляя обратно нотификацию `publishDiagnostics`. Это не «ответ на запрос», это отдельное сообщение от сервера к клиенту.

### Capabilities

В ответе на `initialize` сервер заявляет, что умеет: open/close-уведомления, save-уведомления, completion, hover, и т.д. Neovim читает этот список и решает, что присылать. Заявил `completion: false` — neovim даже пытаться не будет дёрнуть completion. Заявил `save: { includeText: false }` — neovim пришлёт уведомление о сохранении, но без содержимого файла (мы прочитаем с диска сами).

Это полезно: M1 умеет мало, и мы это честно заявляем — не получим запросов, на которые не можем ответить.

### Diagnostics

Это «отчёт о проблемах в файле». Сервер отправляет нотификацию `textDocument/publishDiagnostics`:

```jsonc
{
  "uri": "file:///home/oleg/notes/x.md",
  "diagnostics": [
    {
      "range": {
        "start": { "line": 4, "character": 12 },
        "end":   { "line": 4, "character": 12 }
      },
      "severity": 2,            // 1=Error, 2=Warning, 3=Info, 4=Hint
      "source":   "mdx",
      "message":  "broken link: ./missing.md"
    }
  ]
}
```

Каждая публикация **полностью замещает** предыдущую для того же URI. Очистить — отправить пустой массив. Никаких «дельт» нет.

### URI vs. путь файла

В LSP файл идентифицируется не путём, а URI: `file:///home/oleg/notes/x.md`. На входе мы декодируем URI в путь файловой системы (`/home/oleg/notes/x.md`), на выходе — обратно. Это делает stdlib-пакет `net/url`, отдельной магии не надо.

## Что входит и что не входит

### Входит

- Подкоманда `mdx lsp` — entry point LSP-сервера, общается через stdin/stdout.
- Реакция на жизненный цикл: `initialize`, `initialized`, `shutdown`, `exit`.
- Реакция на `textDocument/didOpen` (открытие файла в редакторе) и `textDocument/didSave` (сохранение).
- На каждое из двух — пере-индексация файла (запись в `notes`/`links`/`tags`) и публикация diagnostics для битых ссылок.
- Один общий код «индексировать один файл», который зовут и CLI (`mdx scan`), и LSP. Это единственная причина, почему в M1 появляется новый пакет `internal/index`.
- Логирование сервера в файл (`$XDG_STATE_HOME/mdx/lsp.log`).
- Юнит-тесты на хендлеры и smoke-тест на полный round-trip через пайпы.
- Раздел в `lsp/README.md` про запуск из neovim и пример настройки `lspconfig`.

### Не входит (отложено)

- `textDocument/didChange` — обновление БД по каждому нажатию клавиши. На M1 — только на open/save, этого хватает для MVP. Добавим позже без поломки протокола: в `initialize` мы изначально декларируем только то, что реально делаем.
- Completion (M3), code actions (M7), workspace/symbol (M4).
- Watcher для файлов, которые редактируются мимо neovim (это работа `mdx gc` в M6).
- Проверка якорей (`#section`) внутри `.md`. Считаем валидной любую ссылку, у которой существует целевой файл.
- Прогрев БД на старте сервера. Полное сканирование запускается руками через `mdx scan`.
- Параллельная обработка нескольких сообщений: на M1 одна большая блокировка вокруг индексации. Не больно для редактируемых вручную файлов; снимем по необходимости.

## Технические решения, которые надо зафиксировать перед стартом

| Вопрос | Решение и обоснование |
|---|---|
| Чем читать/писать LSP-сообщения | Берём библиотеку **`github.com/tliron/glsp`**. Она делает за нас три скучные вещи: (1) фрейминг `Content-Length`, (2) разбор JSON-RPC, (3) маршрутизацию по имени метода в нужный обработчик. Плюс — даёт типизированные структуры всех LSP-сообщений (`InitializeParams`, `Diagnostic`, и т.п.), не пишем свои. Альтернатива «с нуля поверх `jsonrpc2`» возможна, но это +500 строк boilerplate, не относящегося к делу. |
| Транспорт | stdio (`os.Stdin`/`os.Stdout`). По умолчанию `lspconfig` ждёт именно это. TCP/socket — лишняя свобода без пользы. |
| Версия LSP | `protocol_3_16` — то же, что и `lspconfig` использует по дефолту. Это просто номер диалекта; в glsp выбирается импортом нужного подпакета. |
| Где живёт логика «обработать один файл» | В новом пакете `internal/index`. Туда уезжают `scanOne` и `resolveTarget` из `internal/cli/scan.go`. CLI и LSP импортируют один и тот же пакет — единая точка истины. |
| didOpen vs. didSave: где брать содержимое | На didOpen — из payload (`params.TextDocument.Text`). Это содержимое буфера на момент открытия, может отличаться от диска. На didSave — читаем с диска (файл только что записан, диск канонический). |
| Проверка ссылки на «битость» | `os.Lstat(targetPath)`. БД для этого не используем — она может отставать (например, пользователь только что создал целевой файл). |
| Severity для битой ссылки | `Warning`. Не ошибка markdown'а — файл валиден, ссылка не разрешается. |
| URI ↔ путь | Только `file://` URI. `net/url` + `filepath.FromSlash`. Не-`file` URI — игнорируем с записью в лог. |
| Куда писать логи сервера | `$XDG_STATE_HOME/mdx/lsp.log`. Структурированно через `log/slog` (JSON-handler). Уровень `INFO` по умолчанию, `DEBUG` через env `MDX_LOG=debug`. |
| Конкурентность | Один глобальный `sync.Mutex` вокруг секции «прочитать/записать в БД + опубликовать diagnostics». M1 не требует большего; снимем, когда профиль покажет горлышко. |

## Шаги выполнения

Шаги упорядочены так, чтобы после каждого можно было что-то осмысленно проверить руками или тестом.

### Шаг 1. Выделить пакет `internal/index`

Сейчас функция «обработать один markdown-файл» (`scanOne`) живёт внутри `internal/cli/scan.go` и доступна только CLI. На шаге 5–6 мы будем звать её из LSP-хендлеров. Чтобы LSP не импортировал `cli` (это вывернутая зависимость — LSP не должен знать про CLI), выносим логику в отдельный пакет.

Что делаем:
- Создаём `internal/index/index.go` с двумя экспортируемыми функциями:
  - `IndexFile(ctx, conn, path) (Result, error)` — читает с диска, индексирует.
  - `IndexBytes(ctx, conn, path, content) (Result, error)` — индексирует уже прочитанные байты. Это нужно для didOpen, где байты буфера приходят в payload.
- `IndexFile` — тонкая обёртка: `os.ReadFile` + вызов `IndexBytes`.
- `Result` минимальный: `Path` + `Links []ResolvedLink`. Этого хватит для построения diagnostics.
- `internal/index/resolve.go` — переносим `resolveTarget`.
- `internal/index/index_test.go` — переносим `TestResolveTarget`.
- В `internal/cli/scan.go` цикл `Run` теперь зовёт `index.IndexFile`. Локальные `scanOne` и `resolveTarget` удаляются.
- `internal/cli/scan_test.go` удаляется (его тест уехал в `index`).

**Готовность:** `go test ./...` зелёный, `mdx scan` ведёт себя как раньше (e2e-тест в `internal/cli` это и проверяет — он гоняет через `cli.Run`, который теперь делегирует в `index`).

### Шаг 2. Подключить glsp как зависимость

Прежде чем писать сервер, проверим, что библиотека стягивается и компилируется.

Что делаем:
- `go get github.com/tliron/glsp@latest`
- `go get github.com/tliron/glsp/server@latest`
- В `internal/lsp/server.go` пишем функцию `New(conn *sql.DB) *server.Server`, которая возвращает сервер с пустым набором хендлеров. Внутри использует `protocol_3_16.Handler{}`.
- Пока ничего не делаем с этим сервером — просто проверяем, что код собирается.

**Готовность:** `go build ./...` собирает, `go vet ./...` чист.

### Шаг 3. Подкоманда `mdx lsp` и логирование

Cobra-команда, которая открывает БД, поднимает логгер и запускает сервер на stdio.

Что делаем:
- `internal/lsp/log.go::Init(path string) error` — открывает файл, кладёт `slog.JSONHandler` в `slog.Default`. Уровень — `INFO`, `DEBUG` если `MDX_LOG=debug`. Если каталог не существует — создаёт.
- `internal/cli/lsp.go::Run(ctx, conn, logPath) error` — вызывает `log.Init`, делает `New(conn)`, вызывает `srv.RunStdio()`. Блокируется до закрытия соединения.
- В `cmd/mdx/main.go` регистрируем cobra-команду `lsp`. Флаги: `--db` (как у `scan`), `--log` (путь к лог-файлу, дефолт — `$XDG_STATE_HOME/mdx/lsp.log`).
- Резолвинг пути к лог-файлу: env `XDG_STATE_HOME` → `$HOME/.local/state/mdx/lsp.log`.

**Готовность:** `./bin/mdx lsp` запускается, висит в ожидании ввода в stdin. В лог-файле появляется строка `server starting` с путями к БД и к самому логу. Закрытие stdin (Ctrl+D) → graceful exit без паники.

### Шаг 4. Хендлеры жизненного цикла

Сервер должен правильно отвечать на `initialize`/`initialized`/`shutdown`/`exit`. Без этого neovim не считает соединение установленным.

Что делаем:
- `internal/lsp/handlers.go`: четыре функции, навешиваются на поля `protocol_3_16.Handler{Initialize: ..., Initialized: ..., Shutdown: ..., Exit: ...}`.
- `onInitialize` возвращает структуру с capabilities:
  ```go
  TextDocumentSync: TextDocumentSyncOptions{
      OpenClose: ptr(true),
      Change:    ptrSyncKind(None),                // didChange не поддерживаем
      Save:      &SaveOptions{IncludeText: ptr(false)},
  }
  ```
  и поле `ServerInfo{Name: "mdx", Version: ...}`.
- `onInitialized` — пустой, только лог.
- `onShutdown` — закрывает БД, выставляет внутренний флаг «нам сказали shutdown».
- `onExit` — если флаг выставлен → `os.Exit(0)`, иначе → `os.Exit(1)` (LSP-spec требует именно так).

**Готовность:**
1. Юнит-тест: вызываем `onInitialize` напрямую с фиктивным `params`, проверяем, что capabilities те, что ждали.
2. В neovim с настроенным `lspconfig` команда `:LspInfo` показывает `mdx` среди attached серверов. В лог-файле — успешный handshake.

### Шаг 5. Хелпер URI ↔ путь

Маленький, но без него хендлеры didOpen/didSave не напишутся.

Что делаем:
- `internal/lsp/uri.go::URIToPath(uri string) (string, error)` — `url.Parse`, проверка `scheme == "file"`, возврат `filepath.FromSlash(u.Path)`.
- `PathToURI(path string) string` — обратное: `&url.URL{Scheme: "file", Path: filepath.ToSlash(path)}.String()`.
- Тесты: `file:///home/x/y.md` ↔ `/home/x/y.md`, на `http://...` — ошибка.

**Готовность:** unit-тесты зелёные.

### Шаг 6. Хендлер didOpen

Когда пользователь открывает `.md` — переиндексируем содержимое.

Что делаем:
- `onDidOpen(ctx, params)`:
  1. Достать URI и текст из `params.TextDocument`.
  2. URI → путь.
  3. Под глобальным мьютексом: `index.IndexBytes(ctx, conn, path, []byte(text))`.
  4. Из `Result.Links` собрать diagnostics (см. Шаг 8).
  5. Отправить нотификацию `publishDiagnostics` (через `glsp.Context.Notify`).
- Юнит-тест: подсунуть in-memory SQLite, вызвать `onDidOpen` с фикстурным URI и текстом, проверить, что (а) в БД появилась запись `notes`, (б) была отправлена нотификация с ожидаемыми diagnostics.

**Готовность:** при `:edit some.md` в neovim — в `:LspLog` виден didOpen, в `:diagnostics` — те битые ссылки, что есть в файле.

### Шаг 7. Хендлер didSave

Когда пользователь сохраняет — то же самое, но с диска.

Что делаем:
- `onDidSave(ctx, params)`: те же шаги, но через `index.IndexFile` (читает свежую версию с диска).
- Тест аналогичный.

**Готовность:** правка-сохранение `.md` в neovim → diagnostics обновляются.

### Шаг 8. Сборка diagnostics

Превращаем `Result.Links` в массив `protocol.Diagnostic`.

Что делаем:
- `internal/lsp/diagnostics.go::Build(links []index.ResolvedLink) []protocol.Diagnostic`.
- Для каждой ссылки: `os.Lstat(link.TargetPath)`. Если `os.IsNotExist(err)` → диагностика:
  ```go
  protocol.Diagnostic{
      Range:    rangeAt(link.Line, link.Col),    // однопозиционный range на M1
      Severity: ptr(protocol.DiagnosticSeverityWarning),
      Source:   ptr("mdx"),
      Message:  fmt.Sprintf("broken link: %s", link.RawTarget),
  }
  ```
- Range на M1 — без ширины (`start == end`); полную ширину «токена ссылки» считаем позже, когда парсер будет отдавать конец токена.
- Тест: фикстура с битыми и валидными ссылками, проверяем количество и тексты.

**Готовность:** unit-тесты зелёные.

### Шаг 9. Глобальный мьютекс и graceful shutdown

Соберём это вместе и закроем «дырки на выходе».

Что делаем:
- В `lsp.Server` (или внутри замыканий хендлеров) — `sync.Mutex`. Берётся в начале `onDidOpen`/`onDidSave`, отпускается в конце.
- В `cmd/mdx/main.go` для команды `lsp` — `signal.NotifyContext` на `SIGINT`/`SIGTERM`. Отмена контекста инициирует завершение `RunStdio` и закрытие БД.

**Готовность:** `kill -INT $(pgrep mdx)` → процесс выходит, в логе строка `server stopped` без паники.

### Шаг 10. Smoke-тест round-trip

Юнит-тесты проверяли каждый хендлер по отдельности. Smoke-тест проверяет всю цепочку — от приёма JSON-RPC сообщения до отправки ответа.

Что делаем:
- В `internal/lsp/handlers_test.go` (или отдельном `e2e_test.go`):
  - Создать пару `io.Pipe`, передать как stdin/stdout серверу.
  - Записать в stdin серию сообщений: `initialize` → `initialized` → `didOpen` (с фикстурой из M0) → `shutdown` → `exit`.
  - Прочитать с stdout: должны прилететь `InitializeResult` (response) и `publishDiagnostics` (notification).
  - Распарсить, ассертить.
- На фикстуру с битой ссылкой ожидаем 1 diagnostic; на фикстуру с валидной — 0.

**Готовность:** `make test` зелёный, тест воспроизводимый.

### Шаг 11. Документация

В `lsp/README.md` секция «LSP»: как запускать, где логи, минимальный snippet `lspconfig`:

```lua
require('lspconfig.configs').mdx = {
  default_config = {
    cmd       = { 'mdx', 'lsp' },
    filetypes = { 'markdown' },
    root_dir  = function() return vim.loop.cwd() end,
  },
}
require('lspconfig').mdx.setup{}
```

Указать, что сервер обновляет БД только на open/save, а для batch-индексации существует `mdx scan`.

**Готовность:** README покрывает запуск из neovim.

## Критерии готовности M1

1. `make build` собирает бинарь.
2. `make test` зелёный, включая smoke-тест round-trip.
3. `mdx lsp` запускается; в neovim с настроенным `lspconfig` команда `:LspInfo` показывает `mdx` среди attached серверов.
4. После `:edit some.md` (где есть битая ссылка) в `:diagnostics` появляется warning с источником `mdx` и месседжем `broken link: <target>`.
5. После `:write` запись в БД для этого файла обновлена (проверяется sqlite-клиентом или повторным `mdx scan` без diff'а).
6. CLI `mdx scan` всё ещё работает и даёт тот же результат, что и в M0.
7. Лог-файл создаётся, содержит структурированные записи; в stderr сервера ничего не пишется.
8. Закрытие neovim → процесс `mdx lsp` завершается (через `shutdown` + `exit` от клиента).

После этих пунктов M1 закрыт, переходим к M2 (neovim-плагин: навигация).

## Что нужно подтвердить, прежде чем стартовать

1. **glsp vs. сделать всё руками поверх `jsonrpc2`.** По умолчанию — glsp. Влияет на Шаги 2, 4, 6, 7, 10.
2. **Severity для битой ссылки** — `Warning` (предложено) или `Error`. От этого зависит, насколько громко проблема видна в `:diagnostics`.
3. **Куда писать лог по умолчанию** — `$XDG_STATE_HOME/mdx/lsp.log` (предложено по XDG-spec) или рядом с БД в `$XDG_DATA_HOME/mdx/lsp.log`.
