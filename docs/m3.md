# M3 — picker и вставка ссылок

## Цель

После M3 в markdown-буфере нажатие `<leader>mi` (или выполнение `:MdxInsertLink`) открывает Telescope-picker со списком всех известных системе заметок. Источник — SQL-запрос к БД через кастомный LSP-метод `mdx/listNotes`. Выбор заметки вставляет в текущую позицию курсора готовую markdown-ссылку вида `[Заголовок заметки](~/относительный/от/HOME/путь.md)` (или абсолютный путь, если заметка не под `$HOME`).

После M3 пара «создать ссылку → перейти по ссылке» закрыта целиком в neovim, и работает поверх **всей файловой системы**, без понятия «vault» — ровно как описано в стратегии. Picker переиспользует индекс, который наполняет M0 (`mdx scan`) и поддерживает M1 (LSP upsert на open/save). На M4 поверх того же транспортного слоя `mdx/*` добавится поиск по тегам, на M5 — `:MdxRename`.

## Что входит и что не входит

### Входит

- Кастомный LSP-метод `mdx/listNotes` на стороне Go-сервера: возвращает массив записей `{ path, title }` из БД, отсортированный по `title`.
- Функция `db.ListNotes` в Go-слое БД: выборка `path`, `title` из таблицы `notes` через подготовленный запрос.
- Lua-функция `notes.list()`: синхронный JSON-RPC-вызов к подключённому `mdx`-LSP-клиенту через `vim.lsp.buf_request_sync`, возвращает массив записей.
- Lua-функция `insert.format_link(path, title)`: формирование строки `[title](~/...)` для путей под `$HOME`, либо `[title](/abs/path)` для остальных.
- Lua-функция `insert.at_cursor(string)`: вставка строки в позицию курсора через `vim.api.nvim_put`.
- Telescope-picker (`picker.open(opts, on_select)`): источник — `notes.list()`, display — `Заголовок (относительный/путь)`, ordinal для fuzzy-сортировки — `title path`.
- Команда `:MdxInsertLink`, keymap `<leader>mi` (default), регистрация в `setup()`.
- Расширение `setup(opts)`: keymap для `insert_link`. Опции `root`/`exclude`/`link_text` **не вводятся** — за источник заметок отвечает БД, не Lua-плагин.
- Go-тесты на `db.ListNotes` и LSP-метод `mdx/listNotes`.
- Lua-тесты на `format_link`, `at_cursor`, smoke на `insert_link` через подмену `notes.list`.

### Не входит (отложено)

- Создание новой заметки прямо из picker'а, когда в БД нет подходящего файла, — отдельная фича, кандидат на M3.5.
- Поиск по тегам, команда `:MdxTagSearch` — это M4 (поверх транспорта `mdx/*` добавляется метод `mdx/listTags` или похожий).
- Обновление существующих ссылок при перемещении файла (`:MdxRename`) — это M5.
- Section-anchors в формируемых ссылках (`[text](file.md#section)`).
- Автокомплит ссылок при наборе `[` через `nvim-cmp` или встроенный `omnifunc` — это самостоятельный сюжет, может выйти как M3.5 или быть встроен в M4.
- Конфигурируемый формат пути (опция «всегда абсолютный», «всегда `~/...` даже вне `$HOME`») — стратегия упоминает «правило конфигурируемо», но на M3 фиксируем дефолт. Опция — отдельный задел.
- Альтернативные picker-движки (`fzf-lua`, `snacks.picker`, `vim.ui.select`) — поддерживается только Telescope.

## Технические решения

| Вопрос | Решение |
|---|---|
| Источник заметок для picker'а | Кастомный LSP-метод `mdx/listNotes` к серверу `mdx`. Метод не принимает параметров (запрос «выдай все заметки»), возвращает массив `{ path: string, title: string }`. Сортировка по `title` — на стороне Go. |
| Имя метода | `mdx/listNotes`. Префикс `mdx/` — кастомный namespace, отделяет от стандартных LSP-методов (`textDocument/*`, `workspace/*`). Camel-case — соответствует общей конвенции LSP. |
| Транспорт | Стандартный JSON-RPC через stdio LSP-канал. Никаких отдельных сокетов и unix-domain. Lua пользуется встроенным LSP-клиентом neovim (`vim.lsp.buf_request_sync`). |
| Sync vs async LSP-запрос | Sync с `timeout_ms = 2000`. Picker не открывается до получения ответа. БД на старте маленькая (десятки–тысячи заметок), запрос мгновенный; асинхронность с прелодером — преждевременная оптимизация. |
| Регистрация кастомного метода в glsp | Через механизм glsp v0.2: метод добавляется в `protocol.Handler` как поле `CustomRequest`-карты или через обёртку `handler.Methods`. Точная форма — на этапе реализации Шага 2; ключевая инвариантность — JSON-RPC-роутинг по строковому имени метода. |
| Форма Go-структур | `type NoteEntry struct { Path string \`json:"path"\`; Title string \`json:"title"\` }` — публичная, экспортируется LSP-методом и используется как row для `db.ListNotes`. |
| `title` в БД | Уже наполнено в M0/M1 при scan/upsert (первый `# H1` или basename без `.md`). M3 не пересчитывает заголовки — полагается на колонку `notes.title`. |
| Формат пути в ссылке | `~/...` если `vim.fs.normalize(path)` начинается с `vim.fn.expand("~")`; абсолютный путь иначе. Никакой относительной арифметики (`../`/`./sub/`) — стратегия предусматривает стабильность ссылок относительно `$HOME`, а не относительно исходного файла. |
| Default keymap | `<leader>mi` в markdown-буферах. Регистрируется в `ftplugin/markdown.lua` на основе `M.config.keymaps.insert_link`. |
| Скоуп активации команды | Команда `:MdxInsertLink` доступна везде после `setup()`, но фактическая работа возможна только при подключённом `mdx`-LSP-клиенте. Если клиент не найден — `vim.notify("mdx: LSP client not attached", ERROR)`, picker не открывается. |
| Picker engine | Telescope. Зависимость объявляется в lazy-spec `nvim-dev/lua/plugins/mdx.lua`. |
| Тестовая стратегия Go | Стандартные `*_test.go`: `db.ListNotes` тестируется на in-memory SQLite с предзаполненными `notes`-строками. LSP-метод тестируется через `e2e_test.go` (по аналогии с уже существующим в `internal/lsp/`) — отправка JSON-RPC-запроса серверу, проверка ответа. |
| Тестовая стратегия Lua | `format_link` и `at_cursor` юнит-тестируются. `insert_link` покрывается smoke-тестом через подмену `notes.list` (по аналогии с интеграционным тестом из ранее отменённой версии M3). Telescope-UI не тестируется. |

## Структура каталогов проекта (что добавляется к M2)

### Go-сторона (lsp/)

```
lsp/internal/db/
└── queries.go                         ← добавляется функция ListNotes

lsp/internal/lsp/
├── handlers.go                        ← добавляется handler onListNotes
├── handlers_test.go                   ← добавляются тесты на onListNotes

Точное место регистрации кастомного метода зависит от структуры `glsp.Handler`-конструктора в `server.go`; план детализирует это в Шаге 2.

### Lua-сторона (nvim/)

```
nvim/lua/mdx/
├── notes.lua                          ← новое: list() через LSP
├── insert.lua                         ← новое: format_link, at_cursor
├── picker.lua                         ← новое: telescope-обвязка
├── init.lua                           ← обновление: команда :MdxInsertLink, дефолты, M.insert_link
├── link.lua                           (без изменений)
└── resolve.lua                        (без изменений)

nvim/ftplugin/
└── markdown.lua                       ← обновление: keymap для insert_link

nvim/tests/
├── insert_spec.lua                    ← новое
├── link_spec.lua                      (без изменений)

`tests/notes_spec.lua` не создаётся: единственная Lua-функция в `notes.lua` (`list()`) — обёртка над LSP-вызовом, мокировать LSP-клиент в headless-плёнэри неоправданно сложно. Контракт `notes.list` проверяется через Go-тесты на серверной стороне и smoke-тест на интеграции `insert_link`.

## Шаги выполнения

Шаги упорядочены так, чтобы после каждого можно было запустить осмысленную проверку — Go-тест, Lua-тест или ручной сценарий.

### Шаг 1. Скаффолдинг

Создать заглушки на обеих сторонах: Go и Lua. Ничего не работает осмысленно, цель — модули подгружаются и команда регистрируется.

#### Go-сторона

type NoteEntry struct {

func ListNotes(db *sql.DB) ([]NoteEntry, error) {

```

```

Регистрация метода `mdx/listNotes` в роутере — место и форма уточняются в Шаге 2; на этом шаге достаточно добавить пустой handler, чтобы Go-проект компилировался.

#### Lua-сторона

`lua/mdx/notes.lua` объявляет:

```lua

```

`lua/mdx/insert.lua` объявляет:

```

Все функции возвращают пустые значения / `nil`.

В `lua/mdx/init.lua` дописать в `setup()` регистрацию команды:

end, {})
```

И публичную `M.insert_link()` со стабом — `vim.notify("mdx: insert_link not implemented yet", INFO)`.

**Готовность:** `go build ./...` в `lsp/` без ошибок. В neovim `:lua print(vim.inspect(require('mdx.notes')))` показывает таблицу с одной функцией; то же для `mdx.insert` (две функции) и `mdx.picker` (одна функция). `:MdxInsertLink` выводит уведомление-заглушку.

### Шаг 2. LSP-метод `mdx/listNotes`

Реализовать выборку из БД и серверный handler.

#### Состав `lsp/internal/db/queries.go`

`ListNotes(db *sql.DB)` выполняет запрос:

```sql
SELECT path, COALESCE(title, '') AS title
FROM notes
ORDER BY title COLLATE NOCASE, path
```

Возвращает массив `NoteEntry` (порядок по заголовку без учёта регистра, при пустых заголовках — по `path`). При пустой таблице — пустой массив, не `nil`.

#### Состав `lsp/internal/lsp/handlers.go`

`onListNotes(ctx)`:

1. Берёт `s.db` (поле сервера, инициализированное в `onInitialize`).
2. Вызывает `db.ListNotes(s.db)`.
3. Возвращает результат либо ошибку, как есть.

#### Регистрация метода в glsp

В `server.go` (или там, где собирается `glsp.Handler`) добавить кастомный метод. В glsp v0.2 это поле `CustomRequest` map'а или явное расширение через `handler.Methods` — конкретная форма выбирается при реализации, главное — метод резолвится по имени `"mdx/listNotes"`.

#### Тесты

`lsp/internal/db/queries_test.go` — добавить кейсы:
- Пустая таблица → пустой массив.
- Несколько строк → отсортированный массив.
- Записи с пустым `title` оказываются в конце списка.

`lsp/internal/lsp/handlers_test.go` или `e2e_test.go` — JSON-RPC-вызов `mdx/listNotes`, проверка структуры ответа.

**Готовность:** все Go-тесты зелёные. Ручная проверка: `cat | mdx-lsp | jq` или эквивалент с подачей `Content-Length`-обёрнутого `mdx/listNotes` запроса возвращает корректный JSON.

### Шаг 3. Lua-обёртка `notes.list()`

Связать Lua с серверным методом.

#### Состав `lua/mdx/notes.lua`

```lua
function M.list()
```

Поведение:

1. Получает первый клиент `mdx`-LSP через `vim.lsp.get_clients({ name = "mdx" })`. Если клиента нет — выводит `vim.notify("mdx: LSP client not attached", vim.log.levels.ERROR)` и возвращает `nil`.
2. Делает `client:request_sync("mdx/listNotes", vim.empty_dict(), 2000, 0)`.
3. Если результат `err ~= nil` или `result == nil` — выводит уведомление с описанием ошибки и возвращает `nil`.
4. Возвращает `result` (массив `{ path, title }` от Go).

`vim.empty_dict()` нужен, чтобы JSON-RPC увидел пустой объект, а не пустой массив (имеет значение для серверной стороны).

**Готовность:** `:lua print(vim.inspect(require('mdx.notes').list()))` в открытом markdown-буфере выводит массив записей из БД. Если БД пуста — пустой массив. Если LSP-клиент не подключён — уведомление-ошибка и `nil`.

### Шаг 4. Форматирование пути и ссылки

Реализовать правило `~/...` для путей под `$HOME`.

#### Состав `lua/mdx/insert.lua`

```lua
function M.format_link(path, title)
```

Поведение:

1. Нормализует `path` через `vim.fs.normalize`.
2. Получает `home = vim.fs.normalize(vim.fn.expand("~"))`.
3. Если `path` равен `home` или начинается с `home .. "/"` — заменяет prefix на `~`. Иначе оставляет абсолютный путь как есть.
4. Возвращает `"[" .. title .. "](" .. result_path .. ")"`.

#### Тесты

`tests/insert_spec.lua` — табличные кейсы на пары `(path, title, expected)`:

- Файл под `$HOME`: `(vim.fn.expand("~/notes/foo.md"), "Foo")` → `"[Foo](~/notes/foo.md)"`.
- Корневой `~`: `(vim.fn.expand("~"), "Home")` → `"[Home](~)"`.
- Файл вне `$HOME`: `("/etc/hosts", "Hosts")` → `"[Hosts](/etc/hosts)"`.
- Заголовок с markdown-символами (`[Foo](bar)`) — оставляется как есть.

`vim.fn.expand("~")` в тестах динамически портит детерминизм — допустимо, потому что мы только сверяем prefix-замену, а не конкретные значения.

**Готовность:** все табличные кейсы зелёные.

### Шаг 5. Связка picker и `insert_link`

Реализовать `at_cursor`, picker через telescope, оркестрацию `M.insert_link()`.

#### `lua/mdx/insert.lua`

```lua
function M.at_cursor(link_string)
    vim.api.nvim_put({ link_string }, "c", true, true)
end
```

#### `lua/mdx/picker.lua`

```lua
function M.open(opts, on_select)
```

Поведение:

1. `pcall(require, "telescope.pickers")` — если `false`, `vim.notify("mdx: telescope.nvim is required", ERROR)` и выход.
2. Получает `entries = require("mdx.notes").list()`. Если `nil` — выход (notify уже сделан внутри `notes.list`). Если пусто — `vim.notify("mdx: no notes in index", INFO)` и выход.
3. Открывает telescope: `pickers.new`, `finders.new_table` с `entry_maker`, формирующим `{ value = entry, display = string.format("%s (%s)", title, display_path), ordinal = title .. " " .. path }`.
4. На `<CR>`: `on_select(selected.value)`, `actions.close(prompt_bufnr)`.

`display_path` для UI — `~/...`-форма (через ту же логику, что в `format_link`, но без скобок). Это даёт читаемый список.

#### `lua/mdx/init.lua` — обновление `M.insert_link()`

```lua
function M.insert_link()
    require("mdx.picker").open(M.config, function(entry)
        local link = require("mdx.insert").format_link(entry.path, entry.title)
        require("mdx.insert").at_cursor(link)
    end)
end
```

#### Тесты

`tests/insert_spec.lua` — добавить:

- `at_cursor` в пустом буфере, в непустом буфере (как в M2 step 5).
- Smoke-тест `insert_link` через подмену `picker.open`: вместо реального telescope подменяем функцию, чтобы немедленно вызвать `on_select` с заранее заданным entry. Проверяем, что в буфер вставилась ожидаемая строка.

**Готовность:** Lua-тесты зелёные. Ручная проверка: в открытом markdown-буфере `:MdxInsertLink` открывает picker со списком заметок; выбор вставляет строку с `~/...`-путём.

### Шаг 6. Конфигурация и keymap

Расширить `setup(opts)` для управления keymap'ом.

#### Состав `lua/mdx/init.lua`

В `defaults` добавить:

```lua
keymaps = {
    follow = "<leader>mf",
    follow_split = "<leader>ms",
    insert_link = "<leader>mi",   -- новое
},
```

#### Состав `ftplugin/markdown.lua`

После блоков для `follow` и `follow_split` добавить:

```lua
if config.keymaps.insert_link then
    vim.keymap.set("n", config.keymaps.insert_link, function()
        mdx.insert_link()
    end, { buffer = true, desc = "mdx: insert link to existing note" })
end
```

**Готовность:** `<leader>mi` в markdown-буфере открывает picker. `setup({ keymaps = { insert_link = false } })` отключает keymap, но команда `:MdxInsertLink` остаётся работать.

Дописать `nvim/README.md`:

- Команда `:MdxInsertLink`, поведение, требование подключённого `mdx`-LSP-клиента.

## Критерии готовности M3

1. `go test ./...` в `lsp/` зелёный, включая новые тесты на `db.ListNotes` и `mdx/listNotes`-handler.
2. `nvim --headless` с `PlenaryBustedDirectory nvim/tests/` зелёный, включая новые тесты на `insert_spec.lua`.
3. После `mdx scan ~/projects/mdx/` БД содержит как минимум все markdown-файлы проекта; в открытом `.md`-файле `<leader>mi` открывает picker со списком этих заметок.
4. Список в picker'е сортируется по заголовку (без учёта регистра), display — `Заголовок (путь)`.
5. Выбор заметки в picker'е вставляет в позицию курсора строку вида `[Заголовок](~/projects/mdx/...)` с курсором сразу за `)`. Для заметок вне `$HOME` (если такие появятся) — `[Заголовок](/abs/path)`.
6. На отсутствие подключённого `mdx`-LSP-клиента команда `:MdxInsertLink` отвечает уведомлением-ошибкой, picker не открывается.
7. `setup({ keymaps = { insert_link = false } })` отключает keymap, не задевая команду `:MdxInsertLink` и работу `<leader>mf`/`<leader>ms`.
8. README содержит инструкцию по подключению и пример сценария.

После выполнения этих пунктов M3 закрыт; ссылочный цикл «создать → перейти» работает целиком в neovim, на источнике из БД.
